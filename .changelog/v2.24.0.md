## Setup

- **[issue-1792] Windows setup no longer aborts rebuilding esbuild.** Running `setup.ps1` on Windows could stop partway with `Error: Cannot find module ...\client\node_modules\esbuild\install.js`. The script ran esbuild's installer through a hardcoded path that assumes esbuild sits directly under `client/node_modules` тАФ but esbuild is a nested dependency (pulled in by Vite) that npm may place deeper in the dependency tree, so the fixed path was missing and setup bailed. It now rebuilds esbuild with `npm rebuild esbuild`, which finds esbuild wherever it actually landed, matching how the Mac/Linux setup already works. (Closes #1792)
- **[issue-1788] Windows setup no longer fails to launch on the default PowerShell.** Running `setup.ps1` on Windows PowerShell 5.1 errored out before doing anything (`The string is missing the terminator` / `Missing closing '}'`). The scripts contained accented punctuation and an emoji but were saved without a UTF-8 byte-order mark, so the default Windows PowerShell read them in the legacy system codepage and mis-parsed those characters. `setup.ps1`, `update.ps1`, and `scripts/setup-voice.ps1` now carry a UTF-8 BOM so they parse correctly on both Windows PowerShell 5.1 and PowerShell 7+, and a test keeps any future `.ps1` from regressing. (Closes #1788)

## AI Providers

- **[issue-1801] "Test" now finds your CLI provider on Windows.** Testing a CLI/TUI provider (e.g. `claude`) on Windows always failed with `Command 'claude' not found in PATH` even when the command worked fine from a terminal. The test resolved the command with the Unix-only `which` tool, which doesn't exist on Windows тАФ so the check itself errored and was misreported as a missing command. It now uses `where` on Windows (and `which` elsewhere), and runs the version probe against the exact resolved executable, so the test correctly succeeds when your provider is installed. The Test button also now reports **failure** (not a misleading "available") when `where` resolves the command to a Windows `.cmd`/`.bat` npm shim that the `shell:false` agent runner can't actually launch тАФ distinguishing a binary that ran-but-has-no-version-flag (still "available") from one that couldn't be spawned at all. (Closes #1801)
- **[issue-1802] Run local Ollama models that actually edit files with the new "OpenCode Ollama" provider.** Ollama models are trained against the OpenAI tool-calling format, so under the Claude harness they often *describe* a file edit as JSON without ever applying it тАФ the model "responds" but nothing changes on disk. **OpenCode Ollama** drives the `opencode` agent against your local Ollama daemon instead, speaking the OpenAI-compatible protocol natively, so the same local models reliably read, write, and edit files in headless agent tasks. It's disabled by default; enable it, hit **Refresh Models** to pull your tool-use-capable Ollama models, pick a default model, and run. Tool approvals are auto-allowed for unattended execution, and the Ollama connection is configured for you (no `opencode.json` to hand-write). It arrives on the AI Providers page after `./update.sh` (migration `149-opencode-ollama-providers.js` adds it to existing installs) and appears under **Load Samples** on fresh installs. Requires the `opencode` CLI on your PATH. (Closes #1802)
- **[issue-1776] Run CoS agent tasks on a local model with the new "Claude Ollama" provider (CLI + TUI).** If your only LLM is Ollama or LM Studio, you couldn't run file-writing agent tasks тАФ those providers return plain text with no Read/Write/Edit harness. **Claude Ollama** runs the full Claude Code harness on top of your local model: it's a `claude` CLI/TUI pre-wired to your Ollama daemon, and its **Refresh Models** pulls the installed Ollama models filtered to the **tool-use-capable** ones the agent loop depends on. It now ships in **two variants** тАФ **Claude Ollama (local model)** (`cli`, headless) and **Claude Ollama TUI (local model)** (`tui`, interactive) тАФ and **both arrive on the AI Providers page after `./update.sh`** (migration `146-claude-ollama-providers.js` adds them to existing installs and renames the earlier "Clawed Ollama" sample, preserving your settings; previously it lived only in the toolkit sample and never reached the merge path). The task form offers **only coding providers** тАФ raw Ollama / LM Studio / kimi are excluded (with an inline advisory pointing you to Claude Ollama when they're all you have), and a server-side guard rejects an API provider that slips into the spawn path instead of silently writing nothing. Includes a copy-ready global `~/.claude/settings.json` example and a setup guide with the tool-use model caveat. (`data.reference/providers.json`, `server/lib/aiToolkit/`, `scripts/migrations/146-claude-ollama-providers.js`, `docs/features/claude-ollama.md`, `docs/examples/`)

## Scene image rendering

- **[issue-1791] Canceling a scene render now clears its spinner cleanly, and a late older render can't overwrite a newer frame.** Two long-standing rough edges in the background scene-image renderers тАФ storyboard scenes in Writers' Room, catalog portraits, and the Music Video director board тАФ are fixed together. First: canceling a render used to leave its "RenderingтАж" spinner stuck (when the job was still **queued**, which produced no client event at all) or surface a misleading failure (when canceled **while running**, which the generator reported as a failure). Now a cancellation reaches the UI and clears the spinner right away тАФ with no error toast and no error state тАФ everywhere a scene spinner appears, whether the render was queued or already running. Second: if you regenerated a scene frame and an **older** render happened to finish *after* the newer one, it could quietly revert the scene to the stale image тАФ the storyboard and catalog now drop the out-of-order older render (matching the behavior the Music Video board already had), so the newest frame you asked for is the one that sticks (a stale catalog portrait is kept as a reference instead of being lost). Internally the four near-identical media-job completion hooks were consolidated behind one shared tag-dispatch mechanism so this guarantee stays uniform as new render targets are added. (Closes #1791, #1796)
- **[issue-1799] A scene render that fails before it even starts now clears its spinner instead of sticking on "RenderingтАж".** The cancellation fix above handled jobs canceled while queued; this is the failure-side twin. When a media job fails *before* generation begins тАФ for example an unready bring-your-own-video runtime that errors the moment it's dequeued, or a job the watchdog times out тАФ the generator never reached the point of emitting its own failure event, so the scene button could stay stuck spinning forever. Such early failures are now bridged to the same render-failure signal a normal failure uses, so the spinner clears (and the error is surfaced) everywhere a scene spinner appears тАФ Writers' Room storyboards, catalog portraits, and the Music Video director board alike. (Closes #1799)

## Performance

- **Tabbed pages now lazy-load their tab bodies, and `/apps` + `/ambient` moved out of the eager entry bundle.** Digital Twin (17 tabs), MeatSpace (11 tabs), and Brain (10 tabs) previously imported *every* tab component up front, so opening any one tab downloaded a single 240тАУ450 kB chunk's worth of all of them. Each tab body is now a `React.lazy` chunk behind a `<Suspense>` boundary, so a user who views 1тАУ2 tabs no longer pays for the other 15. `Apps` and `Ambient` were the only two non-index pages still imported statically into the entry chunk тАФ they're now `lazyWithReload` like every other route, shrinking the initial bundle (~109 тЖТ ~95 kB gzip). The Vite vendor split also gained named `vendor-three` / `vendor-charts` / `vendor-term` groups so the big 3D/charting/terminal libs land in stable, single shared chunks instead of an opaque `OrbitControls-*.js`. (`client/src/App.jsx`, `client/src/pages/DigitalTwin.jsx`, `client/src/pages/MeatSpace.jsx`, `client/src/pages/Brain.jsx`, `client/vite.config.js`)
- **Pipeline series views read only their own issues instead of scanning the whole table.** `listIssuesForSeries` and `listIssues({ seriesId })` loaded *every* issue in the install, sanitized each, then filtered in JS тАФ wasteful in the autopilot/editorial hot loops that call them per series. They now route through a new indexed `WHERE series_id = $1` query (`idx_issues_series`) on the PostgreSQL backend, with a matching JS-filtered path on the dev/test file backend. (`server/services/pipeline/issues.js`, `server/services/pipeline/issuesStore/{db,store}.js`)
- **Removed avoidable per-render and per-frame work in the client.** `ArcCanvas` rebuilt + re-sorted its seasonтЖТissue buckets (`Set`/`Map`/`sort`) on every render; it's now a `useMemo` keyed on `[seasons, issues]`. `VideoGen` re-filtered the full installed-LoRA list on every keystroke; those two filters are memoized. The CyberCity sky's `useFrame` loop allocated a fresh `THREE.Vector3` every frame (~60/sec) for the sun direction тАФ it now reuses a module-scoped scratch vector. (`client/src/components/pipeline/ArcCanvas.jsx`, `client/src/pages/VideoGen.jsx`, `client/src/components/city/CitySky.jsx`)
- **Collapsed serial awaits and an N+1 insert into batched I/O.** The `/api/apps/:id` detail endpoint read each PM2 process status one-at-a-time тЖТ now `Promise.all`. `dataSync`'s meatspace snapshot and the media-collections ephemeral/refused-parent lookups did serial independent reads тЖТ now concurrent. `memoryDB.updateMemory` inserted one `memory_links` row per related id in a loop тЖТ now a single multi-row `INSERT`. (`server/routes/apps.js`, `server/services/dataSync.js`, `server/services/memoryDB.js`)
