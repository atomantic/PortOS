## AI Providers

- **[issue-1865] CLI provider runs now work on Windows** — npm-installed CLI providers (OpenCode, Codex, Claude Code) ship as `.cmd`/`.bat` shims on Windows; a bare command name like `opencode` never resolves there because `spawn()`'s default `shell:false` doesn't apply Windows' PATHEXT search at all. The Test-provider button surfaced this as "Resolved 'opencode' to ... but it could not be executed" — and that diagnostic was itself hitting a second bug: `where`'s first PATH match can be the extension-less POSIX shell-script stub npm ships alongside the real Windows wrapper (for Git Bash/WSL), which is never launchable natively. A new `resolveWindowsExecutable` (`server/lib/bufferedSpawn.js`) resolves a bare command to its actual `.exe`/`.cmd`/`.bat` path via a deliberately extension-aware filesystem search (skipping that stub). Critically, a resolved `.cmd`/`.bat` target still cannot be launched directly under `shell:false` even with the explicit extension — Node's CVE-2024-27980 patch makes `spawn()`/`execFile()` refuse batch files outright rather than safely auto-wrapping them — so a new `prepareWindowsSafeSpawn` wraps a batch target through Node's own documented-safe pattern, `cmd.exe /c <path> <args>` (a native `.exe` target is launched directly, unwrapped). Because `cmd.exe` is itself a normal native executable, Node's existing, already-tested non-shell argv escaping governs the result — correctly preserving spaces in each argument (including a multi-word vision-captioning prompt) — with none of `shell:true`'s DEP0190 hazard (`shell:true` with an args array does **not** escape arguments, it just space-joins them, so it was rejected as the fix here). This applies to the runner's spawn call, the vendored toolkit's built-in fallback, the autofixer/calendar-sync CLI runner, the vision-captioning CLI runner, and `testProvider()`'s version probe (fixing the Test button's literal reported scenario). Because a `.cmd`/`.bat` target routes through an intermediate `cmd.exe` process, a plain `SIGTERM` only kills that wrapper and orphans the real CLI process; every kill path for these processes (timeout, fallback-signal abort, Stop Run) now uses a `taskkill /T /F`-based process-tree kill on Windows (`killProcessTree`, extended to mark `.killed` synchronously and to skip non-`ChildProcess` killables like a node-pty TUI session, which has its own native teardown) and a plain `SIGTERM` elsewhere, unchanged. ## Architecture

- **[issue-1878] CoS runner now wires the process-level `unhandledRejection`/`uncaughtException` net (defense-in-depth).** The main server (`server/index.js`) already registered a process-level net via the shared `setupProcessErrorHandlers()` helper (`server/lib/errorHandler.js`) — but the standalone CoS runner (`server/cos-runner/index.js`), a separate long-lived PM2 process that spawns and supervises child agents, had none, so a stray async handler that rejected/threw outside the request lifecycle would crash it with Node's default unhandled output. The runner now calls the same shared helper, giving it the identical, already-tested convention: log via the emoji-prefixed `console.error` style, log-and-stay-alive on an unhandled rejection, and log-then-exit-cleanly on an uncaught exception (after flushing, so PM2 restarts a process that would otherwise be left in an undefined state and reaps any orphaned agents on boot). Reusing the shared helper (rather than rolling inline handlers) keeps the two long-lived processes consistent and routes arbitrary non-Error throw values through `normalizeError`, so the net can't itself throw on a `reason` with no `.stack`. This is the backstop for the per-handler try/catch wrapping done in #1825 — not a replacement for it. While wiring this up, hardened the shared helper's `uncaughtException` branch, which dereferenced `error.stack` on the raw thrown value — a non-Error throw (`throw null`) made the net itself throw, masking the original exception and skipping the clean exit/flush; it now guards `error instanceof Error` (mirroring the existing `unhandledRejection` branch). A regression test (`server/process-safety-net.test.js`) asserts the shared helper registers both handlers, that both entry processes wire it up, and that the `uncaughtException` listener tolerates a non-Error throw value. (Closes #1878)

## Security

- **`safeUrlFetch` SSRF guard now pins the vetted IP at connect time, closing a DNS-rebinding TOCTOU** — `isPublicHttpUrlSafe` resolved a hostname's DNS to validate it was public, but the subsequent `fetch` re-resolved the same hostname at connect time, so a rebinding host could answer a public address for the check and a loopback/link-local/cloud-metadata address for the actual connection, slipping past the guard.
