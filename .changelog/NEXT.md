- **CoS investigation tasks now use a human-facing "What happened / What to approve / What unblocks" template with a redacted failure snippet (#2442).** When a CoS agent failure is classified as *actionable*, `createInvestigationTask()` filed a fixed 6-field stub (Failed Agent / Original Task / Error Category / Suggested Fix / Model) that read identically for every category, while the useful diagnostic context stayed buried in `learning.json` telemetry. - **A CoS sub-agent can now challenge a wrong reviewer rejection instead of silently fixing it or getting blocked (#2441).** When a review-loop reviewer rejects work â€” common with noisy local/CLI reviewers â€” the task can be parked in a new `challenged` status with the worker's case attached (`reason` + optional `evidence` + which `reviewer` is disputed) via `POST /api/cos/tasks/:id/challenge`. The full challenge trio (case + resolution + count) rides task metadata, so it round-trips the TASKS.md store, is auditable in the CoS task UI, and federates + converges across sync peers: `challenged` is wired into the claim-aware merge rank (above `in_progress`, below the terminal states), the peer-sync wire status vocab, and a `cosTasks` schema bump (v1â†’v2) so an older peer gently pauses cos-task sync instead of choking on the unknown status. The schema requires exactly one of `outcome` or `recheck`; an unreachable reviewer or unusable findings returns `502 RECHECK_FAILED` rather than silently escalating. ## Fixes

- **Privacy broker case actions are now rendered from the server's state machine, so the UI can't offer an illegal transition.** The drawer and the human-task digest strip used to decide which manual buttons to show from a hand-mirrored client map (`MANUAL_DONE_ACTION`) of the server's `STATE_TRANSITIONS` â€” a copy that could silently drift (the original bug: a "Mark submitted" button on blocked cases, where `blocked â†’ submitted` is not a legal transition, so it always failed). Case payloads and digest items now carry a server-derived `allowedTransitions` list (computed from the same rules `assertTransition` enforces, folding in the any-state `human_task_queued` special case and excluding the rescan-only `confirmed_removed`/`reappeared` targets a human can't initiate). Both action strips render from a shared `state â†’ { target, label, tone }` presentation table (`CASE_ACTIONS`, labels/tones stay UI concerns) filtered by that list, so an action whose target the server doesn't consider legal structurally cannot render.

- **Generated agent prompts (swarm, scheduled claim-work, other queued system tasks) no longer double-print their first line.** A task queued through the CoS improvement/claim-work path is persisted to `COS-TASKS.md` by moving its full multi-line prompt into `metadata.context` and keeping only the first line as `description` (so the markdown stays one-line-per-task). Coming back out, `context` therefore led with a verbatim copy of `description`, and every prompt-render path emitted both â€” so a swarm run opened with the header twice (`# âšˇ SWARM MODE â€¦`, then `### Context`, then `# âšˇ SWARM MODE â€¦` again, the whole body repeated). A new `reconcileSplitContext` step in `agentPromptBuilder.js` detects that exact split signature (context's first non-empty line equals `description`), folds the body back into `description`, and drops the redundant `metadata.context` â€” so the prompt renders once as one clean body with no spurious `### Context` wrapper. A genuinely-separate user-supplied context (whose first line differs from the description) is left untouched, and the stored task keeps its one-line description for the task-list UI. Pure code fix â€” reaches every install on update, no data or prompt-template migration.

- **The CoS agent briefing no longer prints a redundant "Target Application" heading for the PortOS default app on the api-prompt path (#2430).** The light/fallback prompt path already suppressed the target-app line for `portos-default` (the agent runs in the PortOS directory, so it adds nothing), but the editable `cos-agent-briefing` stage template still rendered the heading for any task carrying an app id. The template now gates the stock heading on a precomputed `targetAppLabel` (empty for the default app, the app id for managed apps) that `agentPromptBuilder.js` threads into the render context â€” mirroring `buildTaskBlock`'s predicate â€” while leaving `task.metadata.app` in the context untouched for any custom template references. Ships as stage-prompt migration 187 (hash-matched, cross-synced with 010's accepted-hash chain) so unmodified installs auto-upgrade and customized copies are left alone.

- **Layered Intelligence's outcome feedback loop now reconciles the `plan` (PLAN.md) tracker too (#2435).** After #2428 closed the loop for forge (gh/glab) and jira trackers, the `plan` tracker stayed excluded because `readIssues` mapped every `[lil-*]` PLAN.md item to `state: 'open'` regardless of its `- [ ]` / `- [x]` checkbox, so a completed proposal could never resolve. `extractPlanSlugs` now preserves each item's checkbox state â€” a checked `- [x]` reads `closed` (which `deriveOutcome` maps to *merged*) and an unchecked `- [ ]` (or a bare tag with no checkbox â€” absent â‰  done) reads `open` â€” so a checked item both reconciles as an outcome and falls out of the dedup window to become re-proposable, while open items stay suppressed as before. The `outcomesTrackerSupported()` gate now includes `plan`, so PLAN-tracked apps learn from their own merge rate like forge/jira apps.

- **Character Sheet name edits can no longer be reverted by a stale blur-save that fires while the prior save is still in flight (#2409).** Committing a name change (Enter or blur) starts a PUT and immediately returns the header to its button view. If you reopened the editor before that PUT landed, the reopened input seeded from the pre-save name â€” and a subsequent blur could persist that stale value straight back over the change you just made. The name trigger now disables while a save is in flight (tracked via a `nameSaving` flag set before the PUT and cleared in `finally`) and `startNameEdit` refuses to reopen until the save settles, so exactly one write wins and no stale revert can slip through.

- **HuggingFace model downloads (Flux and every other image/video/music pull) now log to the server console.** Triggering a base-model download showed a progress bar in the UI but left no trace in the server log â€” the entire download path emitted progress *only* over the SSE stream to the browser, and PortOS has no HTTP access logger, so a headless/PM2 log had no record the multi-GB fetch ever ran. The shared SSE driver (`server/lib/sseDownload.js#startHfDownloadStream`, the single chokepoint behind the image, video, and music download endpoints) now writes single-line, emoji-prefixed log lines for the start of each repo pull (`â¬‡ď¸Ź`, with a forced-re-fetch marker), each file as it streams, a cache hit that skips the download (`đź“¦`), the outcome (`âś…` complete with byte count / `âťŚ` failed with the error message), and client-disconnect cancellation (`đź›‘`) â€” while cancelled results are deliberately not logged as failures.

- **Audio â†’ MIDI transcription now prompts for HuggingFace access instead of dead-ending on a gated-repo traceback.** MuScriptor's model weights turned out to live in a *gated* HuggingFace repo (`MuScriptor/muscriptor-*`), so the first transcription on a fresh install â€” even right after the runtime auto-installs â€” 403'd with a raw `GatedRepoError: Access to model â€¦ is restricted` traceback the user couldn't act on. The MuScriptor sidecar now runs under the shared `install_hf_error_handler` (the same cause-chain walker the image/video runners use), so a gated download emits a structured `USER_ERROR:gated_repo:<repo>` marker; the server classifies that into a typed `gated_repo` SSE frame (carrying the repo id), and the client opens the same license-accept + token-paste prompt the gated image models use (reusing `HfTokenBanner`): accept the model's license, paste a HuggingFace read token, and the transcription you originally asked for re-runs automatically. Gated-error detection is now shared via `isGatedRepoError`/`extractGatedRepo` in `server/lib/hfErrors.js` (also adopted by the image runner's prose fallback).

- **Layered Intelligence no longer silently wedges when pinned to an API-only provider.** After LI became an agent-backed scheduled task (#2322), its reasoning run needs a CLI/TUI coding provider (one with a file-writing harness to emit the `.agent-done` sentinel). Installs that carried an HTTP `api` provider (ollama / LM Studio) through migration 184 hit a dead end: every dispatch failed provider resolution *before* an agent registered, so nothing updated the task's status and it sat pending forever â€” silently blocking the app's whole improvement slot. Now the LI `buildTaskInput` hook detects an api-only provider up front and **skips** with an actionable reason (surfaced on the Intelligence tab: *"pick a CLI/TUI provider â€” e.g. the Claude Ollama sample"*), and, as a general safety net, any agent task that fails provider resolution with a **permanent** config error is **blocked** with a `provider-config` reason instead of re-failing on every dispatch.

## Features

- **A challenged CoS task can now be upheld or escalated inline from the task card, and a challenge is bounded by the shared retry budget (#2471).** Follow-up to #2470/#2472 (the challenge-protocol backbone + agent auto-invocation and reviewer re-check). When a sub-agent disputes a reviewer rejection and the task parks in `challenged`, the task card previously only *showed* the worker's case read-only â€” resolving it meant an out-of-band API call. Each unresolved challenge now carries inline **Uphold** (overturn the rejection â†’ re-queue the work) and **Escalate** (let the rejection stand â†’ block + file an arbitration task) controls that call `POST /challenge/resolve` and refresh in place; the buttons disable while a verdict is in flight and disappear once the challenge settles. Separately, the one-shot challenge cap is now unified with the shared `MAX_TOTAL_SPAWNS` retry budget: a task that has already burned its total-spawn budget is refused a challenge (`409 CHALLENGE_BUDGET_EXHAUSTED`) up front, since overturning it would only re-queue work that `agentLifecycle`'s spawn gate immediately re-blocks â€” making the module's "a challenge shares the retry budget" promise real. No migration needed (metadata-only, read on demand).

- **CoS confidence auto-approval now has a safety-kind override, so outward-facing / irreversible work always needs human sign-off (#2440).** Confidence auto-approval used to gate on ONE axis â€” a task type's historical success rate â€” so once a type crossed the threshold, *any* task of that type could auto-run, including inherently outward-facing work: publishing content that goes live, federating records to sync peers, opening external/upstream PRs, or releasing/deploying. A new **safety-kind** axis, orthogonal to confidence, classifies each generated task (`server/services/taskLearning/safetyKind.js`): outward-facing/irreversible kinds (`content`, `publish`, `federation`, `external-pr`) always require the user's approval regardless of how high the success rate is â€” a 100% score can't undo work that has already left the install â€” while reversible internal work (analysis, refactor, code-review, same-repo improvement PRs) keeps today's success-rate gate unchanged. Classification is driven by explicit task metadata (`safetyKind` / `outwardFacing` / capability flags) then conservative keyword signatures, defaulting to *reversible* so existing behavior is preserved unless a task carries an outward signal. The reason is surfaced on the task record (`safetyKind` + `approvalReason`) and in the approval log line, so the UI can show why a high-confidence task still needs review. Tunable via the new `safetyKindApproval` CoS config (`{ enabled, alwaysApproveKinds }`, default on) â€” additive optional key merged on read, no migration needed.

- **The CoS Schedule Timeline can now expand each task track to see and edit its per-app overrides inline.** Which apps run a given scheduled task type (and each app's interval / agent-option / issue-author / swarm overrides) was previously only visible on the Schedule tab's per-task drawer â€” the Timeline showed the task cadence but not who it runs for. Edits save through the existing per-app override endpoints and refresh the timeline in place. System-job tracks (which are not per-app) show no expand toggle. The timeline graph node now forwards the `appOverrides`/`enabledAppCount`/`totalAppCount` plus the global `taskMetadata`/`managedAgentOptions` defaults that `getScheduleStatus` already computes; no migration needed (read-only surfacing of existing data).

- **Layered Intelligence now evaluates a managed app against its OWN goals and performance â€” not PortOS's agent telemetry.** Running the loop on a managed app previously fed it the PortOS install's global autonomous-agent run stats (`data/cos/learning.json`) as the "CoS metrics" source â€” which is on by default for every app â€” so a managed-app run reasoned over PortOS's tooling performance instead of the app's own product performance. Two changes fix this: (1) a new **App metrics (METRICS.md)** telemetry source, on by default for every app, reads a repo-root `METRICS.md` from the app (same as `GOALS.md`/`PLAN.md`) â€” the place an app documents how it measures success (user-success KPIs, production telemetry, where its data lives, and read-only queries an agent can run); and (2) the **CoS metrics** source now defaults **on only for the PortOS install** (where the agent stats genuinely are its own-performance metric) and **off for managed apps** (where they're a tooling/interaction signal, still opt-in).
