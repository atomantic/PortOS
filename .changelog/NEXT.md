# Unreleased

## AI Providers

- **[issue-2336] xAI Grok is now a built-in AI provider — API, Grok Build CLI, and Grok Build TUI.** Three new providers ship ready to enable under AI Providers: **xAI Grok** (the OpenAI-compatible chat API at `https://api.x.ai/v1` — paste your xAI key and pick a model), **Grok Build CLI** (headless, for one-shot generations and file-writing CoS agents), and **Grok Build TUI** (the interactive `grok` coding agent driven over a PTY like Claude Code / Codex). All three arrive disabled by default (nothing calls Grok until you turn one on) and reach existing installs on upgrade. The CLI/TUI use the local `grok` binary; run `grok login` (or set your xAI credentials) to authenticate it.

## Layered Intelligence

- **Layered Intelligence now runs as a visible agent you can watch — and talk to in a TUI — instead of an invisible in-process handler.** It used to be a "handler-backed" task: on each fire it made one inline reasoning call and filed a tracker issue with no agent record, so it never appeared in the CoS queue or the Active Agents page and couldn't be interacted with mid-run. It's now a normal agent-backed scheduled task with two programmatic slots around the agent — a pre-agent step that gathers the app's goals/telemetry/open issues and builds the reasoning prompt, and a post-agent step that validates the agent's structured output and files exactly one deduplicated tracker issue. The reasoning run shows up as an agent like any other automation, and when you select a TUI provider/model you can attach to its terminal and message it while it thinks (`btw`). The "reasoner never writes code" guarantee is preserved by isolation: the agent runs in a throwaway git worktree that is discarded without a commit, push, or merge, so its only durable output is the structured result the filing step consumes. This generalizes into a reusable "programmatic-I/O task" mechanism (pre-agent input hook + post-agent output hook) for future task types that need deterministic data collection or output processing around an agent. Existing installs inherit the throwaway-worktree posture automatically on upgrade (schedule defaults are backfilled) — no migration needed. Per-app enable/interval/provider/model and behavior config (sources, scopes, rules, hand-off) are unchanged, and the Edit App → Intelligence "Last run" status still reflects filed/parked/skipped/no-op outcomes.

- **[issue-2322] Layered Intelligence is now a per-app scheduled task, not a global sweep.** The self-improvement loop used to run as a single global "Layered Intelligence Loop" job that swept every enabled app on one daily fire, with a confusing split between a global on/off (CoS → System Tasks) and per-app config (Edit App → Intelligence) — a per-app card could read "Enabled · Due now" while nothing ran. It now runs like every other self-improvement automation: a per-app, deterministic, handler-backed task in CoS → Schedule with its own per-app enable, interval, provider/model, and "Run now". Enabling the `layered-intelligence` task in the Schedule (plus per app on the Intelligence tab) drives it; the reasoning model still only returns JSON and files one deduplicated tracker issue — no code, no agent. Per-app behavior (telemetry sources, allowed scopes, guidance rules, hand-off) still lives on the app's Intelligence tab. An install upgrade migrates each app's scheduling into its per-app task override and tombstones the old global job, preserving effective on/off state. Note: because it's now a self-improvement task, the loop runs only while CoS improvement is enabled.
  - The standalone `/layered-intelligence` page has been removed — with the loop now a per-app scheduled task, its status is seen and managed in CoS → Schedule like every other task, and per-app behavior stays on Edit App → Intelligence. (The dedicated cross-app overview/proposals page and its endpoints are gone; nothing else surfaced there.)
  - **[issue-2326] The app Automation page now edits per-task provider/model overrides inline.** Each task row on Edit App → Automation gained a "⚙ Configure" disclosure that expands a per-app provider + model override (with "Use default provider" to inherit the global/task default) — previously reachable only from CoS → Schedule → Configure. The `layered-intelligence` row additionally links straight to its behavior config (telemetry sources, allowed scopes, guidance rules) on the Edit App → Intelligence tab, so an app's full self-improvement setup is configurable without leaving the app.

## Added

- **CoS → Timeline: the Workflow tab is now a schedule timeline — see when every active task will run, and fix collisions in place.** The conceptual stage-pipeline visualization at `/cos/workflow` has been replaced with a real execution timeline over a 24-hour or 7-day window: cron launches at their exact slots, interval tasks/jobs at their learning-adjusted next runs and repeats, perpetual tasks as an open-ended draining band with reset/recheck markers, and a "Scheduled order" strip showing the next launches in sequence. Launches from different tasks within 15 minutes of each other get a collision ring plus a "tight handoffs" counter, so schedules can be staggered before they pile onto the concurrent-agent slots. Selecting any track (or an unpinned rotation/on-demand task) opens an inline editor to change its timing mode, frequency, enabled state, weekday-only flag, and hard `runAfter` dependencies without leaving the page — with the zoom window and open track deep-linked via URL params. Projections respect parked app-scoped perpetual tasks and weekday-only schedules (weekend slots that the scheduler would refuse are never drawn), and switching a system job between pinned-cron and interval modes now explicitly clears the inactive mode's fields.

- **[issue-2328] Auto-fix outcomes are now aggregated into dashboard telemetry with a success-rate trend.** The structured `metadata.diagnostics` persisted on every error-driven task (tier / category / failure reason) were queryable but never surfaced. A new pure aggregator (`server/services/autoFixMetrics.js`) reads them off task records and rolls them up by fallback tier, failure category, and task status — plus a time-to-recovery summary (count / avg / median / min / max) and a daily success-rate trend — exposed via `GET /api/autofix/metrics` and rendered by a new **Auto-Fix Telemetry** dashboard widget (overall recovery rate, trend sparkline, per-tier breakdown, median recovery time). To measure time-to-recovery, each diagnostics record now carries an `observedAt` failure timestamp; recovery is derived from the task's completion edit. Sentinel discipline throughout — a rate with no denominator and a duration with no completed-with-timing samples return `null`, never a fabricated `0`, so "no data yet" can't read as "0% success". (Further #2328 slice remains: full 4-tier fallback *execution* wiring — the classifier still annotates the investigation task rather than attempting each tier's fix in sequence.)
- **[issue-2333] Untyped task executions are now auto-classified instead of piling up as `unknown`.** Task-learning's `extractTaskType` used to dump every execution it couldn't pattern-match into a single blind `unknown` bucket — those tasks bypassed routing logic, skipped outcome learning, and leaked unstructured failures. It now runs an execution-entry classification hook (`classifyUntypedTask`, pure/idempotent/<50ms) that, before giving up, infers a concrete domain from additional signals — a `selfImprovementType` metadata hint, an explicit `task.taskType` the primary extractor didn't special-case (e.g. `scheduled` → `scheduled-task`), and broader whole-word description keyword signatures (`fix`/`investigate`/`crash` → `auto-fix`, `refactor`/`optimize` → `self-improve:general`, `review`/`audit` → `idle-review`, `unit test`/`coverage` → `test-task`) — and only falls back to a new sandboxed `external/untyped` type when nothing matches. Newly-classified and fallback executions flow through the existing #2329 failure-signature harvesting so their failures are captured in the normalized telemetry schema. The `external/untyped` bucket is explicitly walled off from routing (it never drives a model-tier suggestion and is never globally skipped) so a heterogeneous grab-bag can't create a routing blind spot or loop. (Partial delivery of #2333; the spawn-time key mirror in `agentModelSelection.extractTaskTypeKey` and deeper self-improvement-loop consumption of the harvested `failureSignatures` remain.)
- **[issue-2333] The spawn-time routing lookup key now mirrors the recorded domain, and the self-improvement loop reads the enriched failure signatures (the #2333 remainder).** Two follow-ons close out #2333. First, `agentModelSelection.extractTaskTypeKey` — the key used at spawn time to look up historical routing performance — no longer falls back to a blind `'unknown'`; it now delegates the fallback to the same `classifyUntypedTask` the completion path records under, so an untyped task's routing lookup keys off the exact bucket its outcome will be recorded in (a concrete inferred domain, or the sandboxed `external/untyped`) instead of a bucket that never matches. Second, a new pure `summarizeFailureSignatures` reader turns the per-completion `failureSignatures` map (provider/model-attributed, success-criteria-aware) into a ranked, display-ready summary that the self-improvement loop now consumes: `getPromptImprovementRecommendations` emits a provider-attributed "route this task type away from X" suggestion (escalated to high priority when recent runs missed a declared success criterion) and exposes the structured signatures, and the Learning insights view surfaces the summary plus a provider/model-named recommendation once a provider crosses three recent failures — attribution the coarse `errorPatterns` count could never express. This closes #2333.
- **[issue-2344] Task-learning now stamps a success-criteria validation verdict on every completion and gates routing aggressiveness on a correlation-quality window (the #2329 remainder).** Each completion now carries an explicit `validationPassed` boolean — "did the run meet its DECLARED success criteria?" — distinct from the runner's exit-code `success`, so a task-referencing-commit task that exits clean but committed nothing is honestly recorded as a miss. Computed once at the shared completion chokepoint (`finalizeAgent`) via the existing `[task-<id>]` commit check and threaded through `buildTaskTelemetryContext`, the persisted agent result, and the harvested failure-signature samples; a null sentinel means "no machine-checkable criterion was declared" (interactive/user tasks, user-terminated runs) so "not declared" never collapses into "declared and failed". Separately, a new rolling correlation-quality window (`server/services/taskLearning/correlationQuality.js`) pairs each run's failure-signal *prediction* (did the enriched signal flag this run's tier, from history before the run) with its actual outcome and scores their correlation via the Matthews/phi coefficient — with a null sentinel (not `0`) for a degenerate window where a class is absent. `suggestModelTier` gates auto-adjustment aggressiveness on that score: until it clears >0.8 with enough samples, tier-avoidance requires a higher failure-sample bar (5 vs 3) so the system doesn't over-correct on an unproven signal. The gauge is surfaced on the Learning insights payload. This closes out the two deferred #2329 slices.
- **[issue-2329] Model routing now consumes the enriched failure signatures, and the latency split derives a queue/compute breakdown when a source timestamp is missing.** The `failureSignatures` telemetry recorded per completion (provider/model/tier-attributed, recency-bounded) is now read by `suggestModelTier`: a new pure `deriveFailureSignalAvoidance` cross-references those recent failures against `routingAccuracy` and steers selection off a tier that is BOTH accumulating recent failures for a task type AND unproven (success rate < 80%, or no success data) — catching a freshly-degrading tier before its all-time rate crosses the hard misroute line, while a proven tier keeps its slot despite absolute failures (signatures record failures only, so counts alone can't condemn a tier). The dominant failing provider/model is surfaced as a `failureSignal` on the suggestion and named in the model-selection log, so the routing decision is legible rather than "avoid tier X". Separately, the queue/compute latency split now falls back to `wall − compute` (or `wall − queue`) when a run carries no `createdAt` or no runner duration, with a per-leg `source` marker (`measured`/`derived`/`null`) so a derived value is never confused with a measured one and a legitimate `0ms` is never confused with an absent leg. (Further #2329 work remains: success-criteria validation booleans on every completion path and the >0.8 correlation-quality measurement window.)
- **[issue-2329] Task-learning now records a structured telemetry context for every completed agent.** On each completion the learning system builds a `TaskTelemetryContext` — a failure signature (`{ category, messageSnippet, failurePosition }`, reusing the category `agentErrorAnalysis` already classified), execution context (`provider`, `model`, `modelTier`, `taskType`, `component`, input-size, and the routing decision rationale), and a latency breakdown (`wallMs` spawn→completion vs. `queueMs` wait vs. `executionMs` measured) — using explicit null sentinels for anything not cheaply derivable rather than fabricating values. Failure signatures are aggregated per error category into a new additive `failureSignatures` block in `learning.json` (older files without the key load unchanged), and the enriched context is surfaced on the single-line completion log. (Partial delivery of #2329; routing auto-adjustment that consumes these signals, success-criteria validation flags, the queue/compute split where spawn timestamps are missing, and correlation-quality measurement remain.)

## Fixed

- **Sidebar nav no longer truncates long link names.** App names (and other child labels) under Apps and the rest of the sidebar used `truncate` in a 224px rail, so names like `barnhub.online` or `IdeatorPro…` collapsed to ellipsis with no way to read them. Labels now wrap with `break-words`, always expose the full name via `title`, the pin control is overlaid so it no longer steals label width, and the expanded sidebar is slightly wider (`w-64`).

- **[issue-2337] Path-configured CLI/TUI providers now resolve their real vendor context window.** `knownProviderContextWindow` (mirrored in `server/lib/stageRunner.js` and `client/src/utils/providers.js`) matched the provider `command` with exact equality, so a Codex/Antigravity/Grok provider configured with an absolute or relative command path (e.g. `/opt/homebrew/bin/grok`) was missed and fell back to the generic 128K CLI default instead of the vendor window (Codex 1M, Antigravity 1.05M, Grok 256K). Both mirrors now normalize the command to its basename before comparing — the server reuses `commandBasename` from `providerModels.js` and the client uses an inline lockstep helper — matching how the arg-builder predicates already route. Low impact in practice (shipped providers carry an explicit `contextWindow`; only hand-configured path-based providers without one were affected).

- **[issue-2328] Auto-fixer now has a circuit breaker so a persistently-failing resource can't flood the task queue.** The auto-fixer only deduped identical errors within a 60-second window, so a provider/model (or critical error) that failed every ~minute would queue a fresh investigation/fix task indefinitely — a runaway loop that exhausts the plan queue and never gets healed by yet another identical task. It now trips a per-resource circuit after more than 3 task-worthy failures within a rolling 1-hour window, suppressing further task creation (logged as `🔌 Auto-fix circuit OPEN …`) until the failure rate ages out and the circuit auto-closes. (Partial delivery of #2328's guardrails; the tiered fallback chain, structured failure-reason telemetry, and dashboard metrics remain.)
- **[issue-2328] Auto-fixer now classifies every failure into a fallback tier and records structured per-attempt diagnostics.** A deterministic classifier maps each error category (config/env, schema/type, transient, or unknown) to one of four fallback tiers — 1 config/env, 2 schema/type, 3 constrained-agent-retry, 4 escalate — with unrecognized categories always escalating so nothing is silently swallowed. Each auto-fix attempt now builds a structured `{ triggerEvent, target, errorType, category, tier, fixStrategy, failureReason }` diagnostics object that rides on the investigation/fix task record (and its agent-facing context) and is surfaced inline in the existing single-line log (`… tier=1 (config/env) …`), so failures can be broken out by tier / category / reason. (Further #2328 slices remain: dashboard telemetry surfacing, time-to-recovery metric, full 4-tier execution wiring.)
- **[issue-2328] Auto-fix diagnostics are now persisted as structured task metadata instead of being silently dropped.** The tier/category/failure-reason diagnostics the classifier builds for every error-driven task were only ever embedded into the free-text context string and the log line — `addTask` never copied the structured object into task metadata, so it couldn't be queried or aggregated afterward. It now persists `taskData.diagnostics` as first-class `metadata.diagnostics` (round-tripping through the markdown store via the existing JSON sentinel, with a defensive non-object guard), and `createCosTaskSchema` validates it for schema parity. This is the prerequisite for the remaining #2328 telemetry slice: aggregating auto-fix outcomes by tier / failure reason / time-to-recovery and surfacing them on the dashboard.
- **[issue-2342] Auto-fix now actually *attempts* the cheapest fallback tier before escalating, instead of only annotating the tier.** The tier classifier (#2331) labeled each provider failure with a fallback tier but the runner only ever did one thing on failure — swap to a fallback provider (Tier 3) — and otherwise queued a Tier-4 investigation task. `runPromptThroughProvider` now walks the tiers in order: a `model-not-found`/`model-not-supported` failure first gets a deterministic **Tier 1 (config/env) correction** — a bounded retry of the *same* provider with a valid model it lists (no persisted config change; skips embedding-only ids) — and only if that declines/fails does it fall through to the **Tier 3** fallback-provider retry, then the **Tier 4** investigation task. Recovery at any tier is reconciled through the existing `noteFallbackStarted`/`noteFallbackHandled`/`noteFallbackFailed` lifecycle, so a Tier-1 recovery suppresses the investigation task exactly like a Tier-3 one — a fully-recovered failure leaves **zero** tasks, and only genuinely unrecovered failures escalate. The recovering tier is surfaced on the result (`fixTier`/`fixStrategy`) and in the single-line log (`🔧 Tier 1 (config/env) retry …`), and because fewer failures reach a Tier-4 task the existing #2328 telemetry automatically reflects the per-tier recovery. (Tier 2 schema/type has no safe deterministic same-provider transform at the runner layer — a malformed request/response can't be rewritten without the caller's schema — so it falls straight through to the bounded retry.)
- **[issue-2350] Tier 2 (schema/type) now performs a real deterministic same-provider request/response correction instead of falling straight through.** Carved out of #2342: at the runner layer a schema/type failure never arrives as a provider *error* (the provider returns HTTP 200), so a malformed response can only be recognized against the caller's declared shape. `runPromptThroughProvider` gained an optional `responseSchema` (a Zod-style schema or a bare predicate) — and an optional `repair` normalizer callback — so a caller can now declare the response it expects. When the response fails that schema the runner attempts a **bounded deterministic JSON coercion in place** (strips markdown fences / prose and pulls the first schema-matching block, reusing `jsonExtract.extractJson`); if the shape still can't be recovered it **re-requests the same provider once with a schema-strengthened prompt** before falling through to the Tier-3 fallback provider and the Tier-4 investigation task. Every recovery is reconciled through the existing `noteFallbackStarted`/`noteFallbackHandled`/`noteFallbackFailed` lifecycle exactly like Tiers 1 and 3, so a Tier-2 recovery leaves **zero** investigation tasks; the recovering path is surfaced on the result (`fixTier: 2`/`fixStrategy: 'schema/type'`, plus `coercedResponse: true` for the pure in-place coercion) and in the single-line log (`🧩 Tier 2 (schema/type) retry …`). A schema/type failure never benches the healthy provider (it's response-specific, not an outage). The Messages triage caller declares its expected JSON-array shape as the first adopter.
- **Layered Intelligence loop now supports every provider type.** The per-app AI-provider picker previously listed only `cli` providers, while the reasoning call went through the api-only `callProviderAISimple` — so a selected CLI provider failed at runtime with "requires an API-based provider," and API/TUI providers (Ollama, LM Studio, Claude/Codex/OpenCode TUI, etc.) never appeared at all. The reasoning call now routes through the unified `runPromptThroughProvider`, which dispatches on `provider.type`, and the picker lists all enabled `cli`/`api`/`tui` providers. CLI/TUI spawns run in the app's repo (`cwd`).
- **A "Run now" of Layered Intelligence now tells you what actually happened.** Because the loop is handler-backed — it files one tracker issue rather than spawning a visible agent — a run that filed nothing gave no usable signal: the on-demand toast collapsed every non-filed outcome (reasoning model returned unparseable JSON, a duplicate/near-duplicate was suppressed, it parked on a blocking issue, no AI provider was configured, Jira wasn't set up, the tracker read failed) into a blanket "nothing to do right now," and on mobile that transient toast was easy to miss entirely. The handler now threads the real terminal reason through the on-demand feedback (so the toast reads e.g. "the reasoning model returned no usable JSON — try a non-reasoning model" or "paused on a blocking issue"), and every run persists its outcome (`lastRunAt` / action / reason / filed ref) so Edit App → Intelligence shows a durable **Last run** status line you can check any time. This is what made a reasoning-style local model (which often emits thinking text with no parseable JSON) look like a queued-but-not-running task with no explanation.
