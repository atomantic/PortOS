## Security

- **Resolved active `npm audit` CVEs in the kokoro-js â†’ protobufjs chain and the client `ws` transitive** â€” the server's `kokoro-js@1.2.1` pulled `protobufjs <=7.6.0` (GHSA-wcpc-wj8m-hjx6 DoS, GHSA-f38q-mgvj-vph7), and the stale `protobufjs: 7.5.9` override sat *inside* the vulnerable range. Pinned `kokoro-js` to `1.2.0` (the audit-stated fix, still within the code's documented `^1.2.x` compatibility window â€” `KokoroTTS`/`TextSplitterStream` exports unchanged) and bumped the `protobufjs` override to `7.6.3` in root + `server/package.json`. On the client, `engine.io-client` nested its own `ws@8.20.1` (GHSA-96hv-2xvq-fx4p memory-exhaustion DoS) that the root `ws: 8.21.0` override couldn't reach; added `ws: 8.21.0` to `client/package.json` overrides. Both `npm audit` runs now report 0 vulnerabilities for these chains. (#1823)
- **Backup restore/delete path validation hardened** â€” `subdirFilter` on `POST /api/backup/restore` was an unconstrained string interpolated into an rsync `--include=${subdirFilter}/***` arg, so `*` could override the filter chain (restoring everything over live data) and `../foo` could traverse out of the snapshot subdir; it now validates against a relative path pattern that rejects wildcard, `..`, and absolute-path segments, with a mirrored defense-in-depth guard in `restoreSnapshot` for non-route callers. `deleteBackup` previously validated `filename.replace(/[.]/g,'')` (so the real dotted filename was never checked, leaving only the `startsWith(backupDir)` guard) â€” it now validates the raw filename directly and explicitly rejects `..`. (#1822)
- **Settings API no longer leaks third-party API tokens** â€” `GET` and `PUT /api/settings` redacted only the top-level `secrets` key, so `imageGen.hfToken` (HuggingFace) and `civitai.apiKey` were returned in full on both initial load and after any save. Both responses now run through a single sanitizer that strips those tokens (sibling config preserved), matching how the Settings UI already reads only token *presence* from the dedicated `/image-gen/setup/hf-token-status` and `/loras/auth/civitai` status routes. On-disk values are untouched. (#1821)
- **Vision-test endpoint no longer reads arbitrary files** â€” `POST /api/providers/:id/test-vision` took `imagePath` straight from the request body, so a `../` traversal or absolute path could have any readable file base64-encoded and forwarded to the configured external vision provider. The image loader now allowlists `imagePath` to a real image basename under `data/screenshots` (via the canonical `resolveScreenshot` path resolver), rejecting traversal and absolute-path escapes before any file is read or forwarded. (#1820)

## Windows updater

- **[issue-1811] Windows update no longer aborts on a successful `git pull`** â€” the PowerShell updater (`update.ps1`) was treating normal git/npm progress messages (which those tools print to stderr, e.g. "From https://github.com/â€¦") as fatal errors and stopping the update partway through. The updater now distinguishes real failures (by exit code) from routine status output, so updates run to completion on Windows. (#1811)

## Fixed

- **Blocking sync I/O removed from request/completion hot paths** â€” several syscalls ran synchronously on paths that serve live SSE/socket traffic, freezing the event loop for their duration. `checkForTaskCommit` (`agentRunTracking.js`) ran a `spawnSync('git', ['log', â€¦])` on the completion path of every failed CoS run (100msâ€“2s on a large repo); it now uses the async, hard-timeout `bufferedSpawn`. `buildId.js` `statSync`'d `client/dist/index.html` on every SPA navigation; the freshness check is now throttled to at most once per second (the hot path between checks is a clock comparison, no syscall), so a rebuild re-syncs the served HTML with the on-disk chunks within the throttle window instead of paying a stat per navigation. The avatar (`routes/avatar.js`) and attachment (`routes/attachments.js`) routes swapped `existsSync`/`statSync` for async `fs.access`/`fs.stat` (via a new canonical `pathExists` helper in `fileUtils.js`), and uploads now call the idempotent `ensureDir` directly. Separately, the tombstone-prune paths for authors/tracks/creative-director projects (DB + file backends) wrapped their per-record `deleteSyncBaseHash` loops in `withBaseHashFlushBatch`, collapsing N `sync_base_hashes.json` rewrites into one. (#1828)
- **External API model/metadata fetches now time out instead of stalling callers indefinitely** â€” `_fetchGeminiModels` (Gemini model refresh in `aiToolkit/providers.js`) made a `fetch` with no `AbortSignal`, unlike every other provider model-fetch in that file, so a hung Google endpoint could block a model-refresh request forever; it now uses the same `AbortSignal.timeout(8000)`. `fetchCivitaiModel` and `searchCivitaiLoras` (`server/lib/civitai.js`) likewise passed no signal, so a Civitai hang stalled LoRA metadata/search callers; both now default to `AbortSignal.timeout(10000)` and accept a caller-provided `signal` override. (#1827)
- **Unguarded async `child_process` handlers can no longer crash the Node process** â€” several `close`/`error` event handlers were `async` and `await`ed disk writes / completion bookkeeping with no `try/catch`, so a single failed write (e.g. a full disk) or a throwing completion hook surfaced as an unhandled rejection and took down the whole server. Wrapped the at-risk handler bodies in `server/services/runner.js` (both `close` and `error`), `videoTimeline/local.js`, `agentCliSpawning.js` (both `close` and `error`), and both the CLI `close` handler and the detached `processStream().catch` in `aiToolkit/runner.js`. The recovery paths now still settle callers waiting on `onComplete` with failure metadata (so a finalization failure surfaces as a failed run instead of hanging forever), with the completion hook isolated from `onComplete` so a throwing `onRunFailed` can't block the caller from settling. Also replaced the two silent recovery-critical `.catch(() => {})` swallows in `cos-runner/index.js` with logged failures so a lost completion-metadata write is visible instead of silently dropping a finished task. (#1825)
- Catalog "Appears in" panel no longer renders dangling chips that deep-link to soft-deleted universe/series/creative-director targets â€” the detail page filters refs to live targets only (the orphan stays recoverable via the "Orphaned" album). Also fixed a stale resolver bug where soft-deleted Creative Director projects (#1564) wrongly resolved as live, and extended orphan-bucket detection so a deleted CD project's ex-cast surfaces as orphaned rather than unlinked. (#1812)
