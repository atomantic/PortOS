# Unreleased Changes

## Character LoRA training

- **[issue-2787] Character LoRA training now sets itself up even when your system Python is too old.** The MLX trainer needs Python 3.10+, and on a machine whose default `python3` is older (or locked down so packages can't be installed into it), the setup step used to fail and leave training unavailable. Now it detects that and builds a dedicated, self-contained Python environment for the trainer instead — and PortOS finds it automatically, so training just works with no Settings to configure. Machines whose system Python already works are unaffected and keep their existing setup.

- **[issue-1227] Character LoRA training is validated end-to-end on Apple Silicon, and now fails fast when it can't run.** A real training run was taken all the way through on the MLX (mflux) runtime — from staging a character's images to a trained adapter, then loading that adapter back into image generation to confirm it renders — so training your own character LoRAs is a proven path on this hardware. The torch-based fallback trainer, meanwhile, can't train on a Mac's GPU (a PyTorch/Metal limitation), so instead of quietly loading gigabytes of weights and then crashing partway in, PortOS now stops you at the start with a clear message to set up the MLX trainer. Nothing changes for Linux/CUDA machines, where the fallback is the intended path.

## Added

- [issue-2763] **Layered Intelligence now reasons from a proposal playbook instead of proposing blind.** Every LI run's prompt now carries a standing **Proposal Playbook** — a distilled rule set for *what* to propose that's present from the very first run, before any per-app outcome data has accumulated. It encodes a Scope Selection Guide (prioritize high-merge scopes like `loop-meta` and `app-data-gap`; only file an `app-improvement` when it clearly serves a named goal), a Success Pattern Catalog and a Rejection Pattern Catalog (a NOT_PLANNED close means a roadmap conflict, so check goals and committed backlog *first*), Task Type Selection Rules (avoid the loop's own chronically-failing self-improvement work; prefer shapes an agent reliably finishes like plan-task / test-coverage / performance), and a Goal Alignment Check against PortOS's core goals and documented non-goals. It's authored as a reviewable markdown file that ships with the code and is injected verbatim into the reasoning prompt; where a live per-app outcome report in the same prompt has real numbers that contradict a general rule, the live data wins. Applies to every app the loop reasons about.

- [issue-2769] **Creative Commissions can now make more than video — pick the output type and the params follow.** A commission's editor gains a **Creative output** selector: choose Video, Image, Music, Music video, or Series, and the generation parameters change to fit (an Image commission asks for an image count and aspect ratio with no duration; Music asks for a length; Series asks for an episode count), instead of always showing video controls. Behind the scenes each type also gets its own tailored brief so the Creative Director is told exactly what to make (produce a still image, compose a music bed, start a new series) and which tools to use — rather than being handed the whole menu and left to guess from your intent. Existing video commissions are unchanged and keep running as before; a scheduled run whose type isn't recognized is skipped rather than mis-made as a video.

- [issue-2757] **The "set your birth date" prompt now tells you when your date is broken rather than just missing.** When your Character sheet or the CyberCity level badges can't show a level, they used to always say "set your birth date" — even if you'd already entered one that turned out to be invalid, dated in the future, or stored in a config file that couldn't be read. Now those cases show a distinct "fix your birth date" prompt (with a warning color) so you're not told to set a date you already entered, while a genuinely unset date still gets the original friendly "set" prompt. Every prompt still deep-links to the age editor where the field lives. The distinction is carried as an explicit, per-machine-derived status that's never persisted or synced between machines.

- [issue-2674] **Your Character sheet now has skills that reflect how you actually use PortOS.** Six skills — Wordsmith (universes, Writers Room works, and catalog records), Mentalist (POST), Vitalist (health logging), Strategist (goal discipline), Archivist (Brain/memory), and Auteur (media) — level up from the work you've already done, on a curve where each doubling of use earns a level, so they climb quickly at first and then ask for real commitment. Nothing new is tracked to make this work: each skill reads the stats its domain already keeps, so the skills are accurate from the moment you upgrade rather than starting everyone at zero. A domain you've never touched is honestly level 0, and a domain whose stats can't be read says "Unavailable" rather than pretending it's zero. Skills are computed fresh on every read and never saved or synced between your machines — usage differs per machine, so syncing them would let the laptop you barely open overwrite the desktop you live in.

- [issue-2675] **Your Character sheet now shows the goals you're actually working toward.** A new Life Goals card sits under your metrics and lists your four most urgent active goals with their progress, each linking straight through to Goals. It's a read-only mirror of the goals you already keep — nothing is stored or duplicated on the Character surface, so there's still exactly one place your goals live and one place to change them. Goals are ranked by the same urgency the Goals page uses; if you haven't set a birth date there's no horizon to rank against, so your goals are listed as-is rather than given a made-up order. Completed and abandoned goals drop off, and if you have no active goals the card invites you to set some instead of rendering an empty block. A card that can't reach your goals says so plainly rather than claiming you have none.

- **Creative Director now has its own Model defaults.** The Creative Director index page gets a **Model defaults** button (deep-linkable at `?models=1`) that pins the provider + model every project uses for Treatment, Production plan, and Scene evaluation — the settings that previously existed only as three rows buried in the global AI Assignments table. Individual projects still override them from their own Models drawer, which now says which default it's inheriting; leaving a stage blank at the Creative Director level falls through to the system default provider, exactly as before. Both surfaces are the same drawer, so the two scopes can't drift.

- **The Brain daily log now saves itself.** Your entry is saved automatically as you type (about a second and a half after you pause, and at least every ten seconds during a long uninterrupted writing run), when you click away from the editor, when you switch tabs or navigate away, and when you background the tab or app — so a closed laptop or a backgrounded phone no longer loses the day. The toolbar shows the current state (`Unsaved…` / `Saving…` / `Saved`) in place of a toast on every autosave; the Save button and ⌘S still work and still confirm with a toast. Autosave never reverts what you're typing mid-save, never writes one day's text into another when you change dates, keeps debouncing instead of retrying per keystroke when the server is unreachable, and deliberately stands down when a dictated voice segment arrives while you have unsaved edits (the toolbar says "Autosave paused — save to keep your edits") — a full-content save would otherwise drop the segment.

- [issue-2701] **Cerebras is now available as an AI provider.** Add your API key on the AI Providers page and PortOS can route AI-backed work through Cerebras's OpenAI-compatible endpoint (very high tokens/sec). Ships disabled with no key, so nothing calls it until you opt in; hit "Refresh models" after adding your key to pull the live model list. Existing installs pick the provider up automatically on restart. Cost estimates on /devtools/usage are also now accurate for `gpt-oss-120b` — it was previously billed at OpenAI GPT-5.4's rates, roughly 10x its real price.

- [issue-2698] **Layered Intelligence now sees the work you've already committed to, and won't propose against it.** A new **Planned work** source feeds the loop your `plan`-labeled tracker issues (or your prioritized Jira backlog, or PLAN.md's unchecked items) alongside the flat open-issue list it used to get, with the instruction to cross-reference its proposal and file nothing when it overlaps something already in scope. It also gets told when its own merge rate falls below 20% and pointed at that backlog as the likely reason — LI's most common rejection was "not planned", which is what filing into work you'd already scheduled looks like from the tracker's side. A tracker that can't be read says so explicitly rather than reading as "nothing is planned", so a transient `gh` blip can't hand the loop a blank cheque. On by default for every app; toggle it off per-app under the app's Intelligence tab.

- [issue-2689] **You can now see how Layered Intelligence's proposals have fared.** The app's Intelligence tab gains a read-only Proposal outcomes panel: your merge rate (of the proposals that have resolved), a breakdown of filed / merged / rejected / abandoned / still-open, why the non-merged ones were closed — grouped by reason using the same diagnosis the loop feeds back into its own prompt — and a list of the most recent proposals with their outcome. It reads only what the loop already records, so it costs no AI calls and no extra tracker lookups; an app that has filed nothing says so plainly, a store that can't be read says "couldn't load" rather than pretending your history is empty, and proposals still awaiting triage are counted as open, never as failures.

- [issue-2700] **Layered Intelligence now checks its own work before it files.** A new **Self-evaluation** source hands the loop a deterministic read of its own record — how many of its proposals you actually merged (and the reasons you rejected the rest), which proposals it has already filed that are still suppressed, and whether its own agent runs are even succeeding — so it can judge a proposal's quality *before* filing instead of only learning from your rejections afterward. When it's reasoning with little evidence about itself it's told to hold a higher bar and file nothing rather than guess; when its own runs are failing more than half the time it's told the problem may be the loop rather than the app, and to stop marking work for unattended hand-off. Every signal is either present or explicitly reported missing: "no proposal has been triaged yet" never reads as "everything I file gets rejected", and a tracker that couldn't be read never reads as "I've filed nothing" (which would license a duplicate re-file). Costs no extra AI calls — it's computed from data the loop already has. On by default for PortOS, off for managed apps; toggle it per-app under the app's Intelligence tab.

## Changed

- [issue-2773] **A scheduled Series commission now actually generates its episodes instead of leaving an empty series.** When the Creative Director plans a Series commission, it first creates the series (which mints a fresh id) and then has to run that new series' autopilot — but until now the plan couldn't refer to an id that didn't exist yet when the plan was written, so a Series commission could only ever produce an empty shell. Plan steps can now reference a value produced by an earlier step with `{{steps.<stepId>.result.<key>}}` (e.g. `{{steps.create-series.result.id}}`), and the server resolves it at dispatch time — so the "create the series, then generate its issues/episodes" hand-off works end to end. A plan that references a step that didn't run or didn't produce the named id fails that step with a clear reason and re-plans rather than dispatching an unresolved placeholder. Existing installs pick up the planner guidance automatically on restart.

- **The Cyber Muse (3D) Chief-of-Staff avatar now works with real color and a livelier "coding" state.** The rigged robot renders in its own full textures and color instead of a dark holographic silhouette with the agent-state hue painted over it — the per-state color now lives in the surrounding halo, ground glow, sparkles, and accent light, so the model reads clearly while the scene still signals what the agent is doing. And while an agent is working, the avatar no longer loops a single punch clip: it cycles a varied montage (jab → run → jump → thumbs-up → walk → celebrate) so a busy Chief of Staff looks busy. The run/walk cycles play "in place" so the avatar can't stride out of frame, and a custom GLB missing the montage clips falls back to its single base loop exactly as before.

- [issue-2677] **Your Character page is now about you, not a generic adventurer.** The page centers on the human who owns the install — your name, your avatar, your age-based level, and the skills, goals, and metrics derived from how you actually use PortOS — with the old D&D combat mechanics moved out of the way. Damage, rest, dice rolls, manual XP, the JIRA/task XP sync, and the event log now live in a collapsed "RPG mechanics" section that's closed by default; nothing was removed, so those buttons and endpoints still work and existing character files keep their history. When you haven't set a birth date yet there's no age to turn into a level, so instead of a blank badge the page prompts you to "set your birth date" and links straight to the age editor where that field lives. Your name and title now read as human placeholders ("Your name", "Add a title") instead of the old generic "Adventurer" / "Developer": a fresh install starts blank so it prompts for your real name (an existing install keeps whatever name is on record — edit it inline to make it yours).

- [issue-2669] **MeatSpace health forms no longer double-toast on failure.** Saving a genome marker note, or adding/editing a custom drink or custom nicotine product, showed two stacked error toasts when the request failed — once from the API helper and once from the form's own catch. These five sites now suppress the helper toast so a single error is shown.

- [issue-2669] **Manuscript editorial fixes and the Local LLM status load no longer double-toast on failure.** Generating or accepting a suggested manuscript fix, and loading the Local LLM settings status, each stacked two error toasts on failure — once from the API helper and once from the caller's own error handler. These three sites now suppress the helper toast so a single error is shown.

- **Clicking a Creative Commission now opens its own page showing every render it has made.** A commission used to open a sidebar edit form; it now navigates to a routed detail page (`/creative-commission/:id`) that leads with a gallery of every render the commission has produced — each run's final video (with an inline play badge), rendered scene, or produced image, newest first — above the full editable brief/schedule/generation config. Rate a render 👍/👎 and add a note right on its card to steer the next run, run the commission now, pause it, or delete it from the header. The render, its Creative Director output, and the open commission are all directly shareable and bookmarkable (the URL is the source of truth), and a notification about a scheduled run deep-links straight to the render. Creating a new commission still uses the quick "New Commission" drawer on the index; a run whose project was pruned degrades to a status-only card with its output link intact.

- [issue-2702] **The Creative Director now shows you what it made.** Project cards on `/creative-director` lead with a preview of the produced media — the final video's poster with a play badge (click the artwork to open the project, the badge to play it inline), a rendered scene, or a produced image from a directive plan — falling back to a tidy "no render yet" placeholder instead of a wall of text. The Overview tab renders the final cut inline instead of showing only a job id, keeping the Media History link alongside it; plans that emit images show the image. Previews respect each project's aspect ratio (16:9 / 9:16 / 1:1), and the grid never downloads a video until you actually press play.

- [issue-2689] **Layered Intelligence now tells its reasoner *why* each recently-rejected proposal was turned down, so it stops re-filing known failure patterns.** The loop's self-check already listed the proposals it filed recently that are still suppressed and warned "do NOT re-propose these" — but it named only their dedup slugs, so the reasoner could avoid an exact duplicate yet re-file the very same idea, worded differently, that had already been declined for being out of scope or missing context. Each named proposal now carries the reason it was closed ("— previously closed: outside the app's scope"), read straight from the rejection diagnosis the loop already records, so the reasoner sees the specific failure pattern and can route around it rather than repeating it. Costs no extra AI calls and no extra tracker reads; a proposal that was closed with no recorded reason is simply left unannotated rather than guessed at.

- [issue-2748] **Layered Intelligence now reads the closing comment to work out *why* a proposal was declined, not just that it was.** GitHub records a rejected proposal only as "closed as not planned" — declined, but with no cause — so the loop's most common rejection reason was the generic "the user declined it", which tells the reasoner nothing to route around. When such a close carries a comment that states the cause ("this is out of scope", "can't reproduce, too vague", "low-quality proposal"), the loop now scans that comment with a deterministic keyword pass and sharpens the generic decline into the specific taxonomy token (out of scope / missing context / quality). A specific signal still wins — an explicit label, or a specific close reason like duplicate, outranks the comment, because free text is noisier than something a human deliberately applied — and the comment only ever refines the *reason*, never the merged/rejected outcome, so it can't move the merge rate. Costs no extra AI calls and no extra tracker reads (the comment text rides along in the issue list the loop already fetches), and nothing about how records are stored changes, so older installs re-diagnose their history on the next run with no migration.

- [issue-2689] **Layered Intelligence now diagnoses *why* its proposals were turned down, instead of parroting the tracker back at itself.** When a filed proposal is closed unmerged, it's classified against a fixed vocabulary — duplicate, declined, out of scope, missing context, quality, environment blocker, and more — read from the issue's close reason and its labels, and stored alongside the outcome. The loop's feedback block used to tell it "common rejection reasons: not_planned", which restates the outcome rather than explaining it; it now reads "already tracked elsewhere (2); the user declined it (1)". A proposal closed with nothing to explain it is recorded as an explicit unknown and counted, so "3 of 4 closed with no recorded reason" is reported to the loop as the blind spot it is — never dressed up as a real diagnosis, and never allowed to read as "nothing was ever rejected". Rejections the loop hasn't classified yet are reported as their own separate count rather than being folded in with the ones it examined and couldn't explain, because the two have different remedies. Merged proposals never acquire a rejection reason, which matters for Jira- and PLAN.md-backed apps whose trackers report no close reason at all and whose ordinary close *is* a merge. Costs no extra AI calls and no extra tracker reads — every reason is derived from issue data the loop already fetches — and rejection history recorded before this upgrade is re-diagnosed on the next run rather than needing a reset.

- Daily log entries now re-embed into the Chief-of-Staff memory index once you settle (coalesced per day, ~30s after your last edit) instead of on every single save. Autosave means a day's entry is now written every couple of seconds while you type; re-embedding per save would have meant an embedding call per typing pause — and, once a day's log grows past the embedding size budget, an unrequested AI summarization call each time too. Search results are unchanged; the index simply updates when you pause rather than mid-sentence, which also produces a better vector than embedding a half-typed sentence a hundred times. Journals now use the same debounced, coalesced re-embed path every other Brain record type already used.

## Internal

- **[issue-1893] Decided and documented `atomicWrite`'s symlink semantics.** Settled the design question deferred from the #1837 review: when `atomicWrite`'s target is a symlink, the temp+rename deliberately REPLACES the link with a regular file rather than following it to the backing file — the standard atomic-write contract (git, dpkg, rsync, `os.replace` all replace the link), and following it would reintroduce the non-atomic in-place truncate the helper exists to eliminate. The decision (and the reasoning behind not following or rejecting) is now recorded in the helper's JSDoc, and a test pins the replace-not-follow behavior so it can't silently regress. No runtime behavior changed — PortOS ships no symlinked data files — this closes the open question rather than altering the writer. (`server/lib/fileUtils.js`, `server/lib/fileUtils.test.js`)

- **[issue-725] Brain entity stores split to per-record `collectionStore` layout.** Every Brain store (people, projects, ideas, admin, memories, links, buckets, journals, inbox, songs) moved off its monolithic `data/brain/<type>.json` file — where every create/update/delete rewrote the whole file and every read parsed all records behind a 2s cache and one global write mutex — onto the per-record `collectionStore` shape (`data/brain/<type>/<id>/index.json`), read through to disk with per-id write queues so writes to different records no longer serialize. This is proactive scale headroom: PortOS is distributed and a user's federated peers accumulate Brain data on their own schedule, so stores are prepared for size before any one install hits the whole-file-rewrite cliff. Federation is untouched — Brain federates strictly per-record through the storage API seams, so the on-disk container shape is orthogonal to the wire format (no schema-version bump, no wire-category change), and in-place tombstones are preserved through the split so no last-writer-wins delete marker is lost. Migration 200 splits existing installs (legacy files renamed to `.bak-200`, never deleted; idempotent and recovery-safe), and the shared split skeleton (`makeSplitMigration`) gained a `map` mode for the object-keyed `{ records: { <id>: rec } }` shape alongside the existing array mode. Every exported `brainStorage` function keeps its signature and behavior, so all runtime consumers and the whole reconcile/anti-entropy path are unchanged. (`server/services/brainStorage.js`, `scripts/migrations/200-split-brain-stores-to-per-record.js`, `scripts/migrations/_lib.js`)

- **[issue-2775] De-flaked the MidiVisualization "Space toggles playback" test.** The MIDI-visualization component tests asserted on the transport button's *optimistic* `playing` state and raced the synth player's async lifecycle (`play()`'s `await ctx.resume()` microtask hop and the immediate `totalSec <= 0 → onEnded` reset), so `findByLabelText('Pause playback')` failed intermittently in CI and passed on re-run. `useMidiPlayer` is now mocked in the component test with a synchronous state-flip stub, so the three chrome-wiring assertions (playing → label, click/Space → toggle, collapse → teardown) are deterministic and no longer depend on audio timing. The hook's real async behavior stays covered directly in `useMidiPlayer.test.jsx` and the player lib in `midiPlayback.test.js`, so no coverage is lost. (`client/src/components/songs/MidiVisualization.test.jsx`)

- **[issue-2706] Stage-seeding migrations now guard their own `data.reference` assets.** Migrations 189 (model-personality), 094 (object-attachment checks), and 182 (CWQE quality) each seed stage prompts on upgrade but shipped without a colocated test, so a rename or removal of a reference template would have surfaced only at runtime as "Stage not found" on a fresh install. Each now has a `.test.js` asserting it seeds when absent, no-ops on re-run, never clobbers a customized template or hand-tuned config entry, and — the point — that every `data.reference/prompts/stages/*.md` template it names and its `stage-config.json` entry actually ship. The guard is anchored to the migration's own stage list (now exposed on the object the seed helper returns), so adding a stage without shipping its template fails the suite instead of leaving it green against a stale hand-copied list. The three suites collapse to one `runSeedStageMigrationTests({ migration, stages, prefix })` call apiece via a new shared helper — the seed-family analogue of `_testHelpers.js`'s `runPromptMigrationTests`. (`scripts/migrations/_seedStageTestHelpers.js`)

## Fixed

- [issue-2772] **A scheduled Music commission now produces a track you can actually play and rate, not just a job id.** When a `music` commission's Creative Director ran, its planner enqueued the audio render but never tagged it for the project — so the finished track was never filed onto the project and the commission's run preview had nothing to show. Planner-enqueued audio jobs are now tagged the same way the first-pass music flow already was, so the durable music-bed hook files the rendered track onto the project the moment it completes (no open browser required), and the commission run preview surfaces it as an inline audio player you can play and thumbs-up/down to steer the next run. Video and image commissions are unchanged.

- [issue-2770] **Losing the migration bookkeeping file can no longer wipe legitimately-earned learning history.** The two one-time cleanups that cleared mis-recorded task stats (#2700, #2696) find their target by presence — "is this bucket here?" — which is exactly right the first time, but dangerous on a rerun. If `data/migrations.applied.json` (the list of which migrations have run) ever went missing or got corrupted, the runner rebuilt it from scratch and reran *every* migration — so those cleanups would fire again and delete the real success/failure history you'd accumulated since. Migrations that purge by presence now declare themselves to the runner, and the runner refuses to rerun one whenever it had to rebuild that bookkeeping file from empty: your post-fix history survives, and on a fresh install (where there's nothing to clear anyway) the cleanup is simply recorded as done. The guard lives in one place and is shared by both cleanups, so they can't drift apart.

- [issue-2696] **Background jobs that finish your work through git/GitHub/Jira instead of a commit no longer report a permanent 0% success rate.** The branch-reconcile, issue-reconcile, branch-cleanup, and Jira status-report jobs deliver their result as a side effect — a merged PR, a resolved conflict, a deleted branch, a posted report — so they deliberately never make a `task-…` commit of their own. But the success check only knew how to look for that commit, so it scored every *successful* run as a failure and pinned those task types at 0% on the CoS Learning card (which in turn steered the self-improvement loop away from them on evidence that was never true). They're now judged the same way pipeline and media jobs already were — by whether the run itself succeeded — and a one-time migration clears the mis-recorded history on upgrade (and the "Backfill History" button no longer re-poisons it from the old records). This is the same measurement bug already fixed for the Layered Intelligence reasoning run; jobs that genuinely do commit their work — the accessibility audit, the Jira sprint manager, the PLAN.md replan — are unaffected and still judged on their commit, so a real failure there is never masked.

- [issue-2737] **An unattended background AI job that hits a failing provider no longer buries you in a stack of red error toasts.** Jobs that run without you watching — like the weekly goal check-in, which makes one AI call per goal at once — used to fire a separate red error toast for *every* failed call when a provider was down (an expired key, a dead endpoint, a model returning nothing). A seven-goal check-in against a broken provider meant seven stacked toasts you never saw coming. Those background failures now collapse into a single toast per provider that counts them ("Example Provider failed on 5 background AI calls"), with the first one still showing the provider's real reason. AI actions *you* triggered are unchanged: they still report their failure the instant it happens, one toast each, with the provider's actual error — so a failure you're waiting on is never aggregated away.

- [issue-2752] **Generating a Character avatar before you've entered your name no longer produces a D&D adventurer.** The Character reframe leaves a fresh install's name and class blank, but clicking "Generate avatar" first still fell back to a fantasy `fantasy portrait of an adventurer, warrior class, D&D character art …` prompt — the exact generic RPG art the reframe moves away from. The avatar fallback is now a human portrait: a blank identity renders a neutral `portrait of a person, photographic, natural lighting …`, and once you've filled in your name (and class) it shapes a photographic portrait from those instead of a fantasy one. Callers that genuinely want a fantasy avatar still pass their own prompt to opt out.

- [issue-2726] **Your Character sheet's Mentalist, Vitalist, and Strategist can now tell an unreadable file from an empty one.** The sheet promises that a domain it can't read says "Unavailable" rather than pretending you've done nothing — but the three skills backed by files (POST practice, health logging, and goals) couldn't keep that promise: the shared file reader returned an empty default for *every* failure, not just for a file that was never written, so a corrupt or unreadable `post-sessions.json` scored a real-looking "Mentalist — Lv 0, 0 logged". Health was the worst case: all five of its logs could fail and still add up to a perfectly plausible zero. Those reads are now strict — a file that's genuinely absent still counts as the honest zero it is, while one that's present but unreadable or corrupt reports as Unavailable, the same way the database-backed skills always have. The Metrics grid gets the fix for free, since both read the same signals. Nothing else changes behavior: every other screen that reads these files still degrades gracefully to an empty list rather than breaking, which is the right call for a view you're using rather than a number you're being told.

- [issue-2700] **Layered Intelligence runs were being recorded as failures even when they succeeded.** An LI run is a *reasoning* run: it's explicitly told not to commit or open a PR, and hands its result back through a sentinel file instead. But the shared success check only knows how to look for a code commit, and its verdict overrides the run's actual exit code — so every correct LI run was filed as a failure, dragging the loop's success rate toward 0% and skewing anything built on it (task routing, the CoS Learning card, duration estimates). Task types that hand back results programmatically rather than as a commit are now exempt from the commit check, the same way pipeline and media jobs already were. Ordinary code tasks are still held to it. Because the mis-recorded history can't be reconstructed after the fact, installs that already ran the loop get that one bucket cleared on upgrade, so it starts from an honest "no runs recorded yet" rather than a fabricated 0% — no other task type's stats are touched.

- [issue-2727] **A Layered Intelligence run that finished cleanly but produced nothing is no longer recorded as a success.** Exempting these reasoning runs from the code-commit check (above) left them judged purely on whether the process exited cleanly — so a run that exited fine but wrote back a missing or unreadable result, or whose follow-up step crashed, still counted as a win and quietly inflated the loop's success rate. These runs are now judged on what they actually hand back: the result was readable and the follow-up step could act on it. A run that returns nothing usable is recorded as the failure it is, a run that delivers is recorded as a success with no commit required, and the "nothing filed this time" outcomes (no proposal, duplicate, out of scope) still count as successes — the run did its job. Output that isn't a real answer at all (a stray sentence where a structured result belongs) is now told apart from a genuine "I looked and there's nothing to propose", instead of both counting as a win. A run whose reasoning was sound but whose issue tracker was simply unreachable still counts as a success — an outage at GitHub isn't the run's fault, and shouldn't drag the loop's score down or pause it. Ordinary code tasks are still held to the commit check, and pipeline and media jobs are unchanged.

- **The Scene evaluation model dropdown no longer comes up empty when you have vision models installed.** Picking Ollama for a Creative Director project's Scene evaluation showed an empty model list (and a bare text box) even with vision-capable models pulled, because the browser decided what counts as a vision model from a hard-coded list of model-name patterns that knew `gemma-3` but not `gemma4` — so `gemma4:e4b` and `qwen3.6:35b` were filtered out as if they were text-only. Vision pickers now ask the server, which reads each model's real capabilities from the backend itself (Ollama's `/api/show`, LM Studio's `vlm` tag), and union that with the old name matching — so a newer model family shows up without waiting for the pattern list to catch up, and nothing that used to be offered disappears. The AI Assignments table's vision rows get the same fix. ## Layered Intelligence

### Changed

- [issue-2760] **Layered Intelligence now steers its proposals toward work it can actually finish.** Each run reads its own track record — which kinds of task it has historically completed versus consistently failed — and hands the reasoner an explicit "prefer these, avoid those" summary before it decides what to propose. So the loop stops repeatedly filing work in areas where its follow-through fails and concentrates on the kinds of improvement it lands reliably. The read is fresh every run, so an area recovers on its own the moment its success rate climbs back — nothing to reset by hand. This only kicks in once an area has enough completed runs to be meaningful, and applies to PortOS's own self-improvement loop (managed apps, which don't track these run stats, are unaffected).

- [issue-2765] **Layered Intelligence's "prefer/avoid" steering now learns from its OWN proposals, not just install-wide task stats.** The earlier steering (above) borrowed how reliably each *kind* of task gets finished on this install — a useful-but-loose signal, since a given proposal doesn't map cleanly onto one task type. Now, when the loop hands one of its own proposals to a coding agent to implement, the outcome of that run is recorded against the proposal's *domain* (the scope it was filed under), and future runs get a direct "here's how your own proposals in each domain actually turned out" summary alongside the older directional one. So the loop can favor the domains where its proposals genuinely land and be more cautious in domains where its hand-offs stall even after the proposal was accepted — a true feedback loop rather than an approximation. This builds up only as the optional hand-off path runs, and never counts a run that failed for an outage or rate limit against a domain.
