# Unreleased

## Changes

- **Media-gen marker parsers converged onto `createLineReader` too (#2463, follow-up to #2392).** The two remaining hand-copied per-chunk splitters — `imageGen/local.js` (stderr + stdout) and `videoGen/local.js` (stderr, plus a stdout `'\n'`-carry that never flushed the last line and decoded each chunk independently) — parsed `STAGE:`/`STATUS:`/`DOWNLOAD:` protocol markers with no carry buffer, so a marker or multibyte char split across a pipe chunk boundary could tear a progress event. Both services now route line parsing through `server/lib/streamLines.js#createLineReader` (StringDecoder-backed, carry across chunks, `flush()` on `close`), while the render-critical machinery is left exactly as-is: idle-stall + completion watchdogs still fire on the same signals (`resetIdleStallTimer` stays in the per-chunk `data` handler; the videoGen flush runs *before* the watchdog clears so a final result-JSON line can't re-arm a stray SIGKILL), the imageGen `stderrBuffer` error tail keeps accumulating raw chunk text, and the stdout JSON `video_path`/result capture is unchanged. Behavior-preserving; adds a chunk-boundary test proving a marker split across two chunks parses as one event.

- **Duplicated child-process stream/discovery helpers converged into shared lib modules (#2392).** Three hand-copied "split a stdout/stderr chunk stream into lines with a carry buffer" implementations (`ytdlpAudioImport.js`, `videoDownload.js`, `hfDownload.js`) had already drifted — only hfDownload's copy flushed the final unterminated line. They now share `server/lib/streamLines.js#createLineReader(onLine, {splitRe?, maxCarry?})`, which carries the partial trailing line across chunks, exposes a `flush()` for the last newline-less line (so the two yt-dlp paths stop dropping it — and skip the flush on cancel so a partial marker can't fire a stray progress frame), clamps the carry buffer against a newline-less runaway stream, and takes a `splitRe: /[\r\n]+/` option for bare-`\r` torch/tqdm progress redraws. The four bare-`\r` progress-splitter siblings (`midiRuntime.js`, `videoGen.js`, `music.js` install-log routes, and `pythonSetup.js#streamSpawn`) are converged onto the same reader with that option, so a progress line split across a chunk boundary is no longer torn. Separately, the inline `which`/`where` PATH probes in `ytdlp.js`, `ffmpeg.js` (×2), `pythonSetup.js`, and voice `bootstrap.js` are replaced by a single `whichFirst(name)` in `server/lib/processEnv.js` (Malloc-stripped env, 5s timeout, first-line result). The vendored `aiToolkit/providers.js` copy is intentionally left alone (self-contained by design). Behavior-preserving; both new helpers are unit-tested.

- **Runtime/venv install SSE flows now log to the server console too, matching the HF model-download logging (#2432).** The image/video/MuScriptor runtime installers (spawning `setup-image-video.sh`), the FLUX.2 venv bootstrap, and the local image-gen pip installer streamed multi-GB pip/git progress only to the browser via SSE — a headless / PM2 log had no record the install ever ran (the same gap `startHfDownloadStream` had already closed for HF model downloads). A new shared `createInstallLogger` chokepoint (`server/lib/installLogger.js`) now emits a single-line START, throttled heartbeats + `stage` milestones, and the OUTCOME (success/fail/cancel) with installer name, target, and elapsed time — without echoing every raw pip/bash line. Wired into all five install spawn sites in `music.js`, `videoGen.js`, `imageGen.js` (FLUX.2 venv + local pip install), and `midiRuntime.js`.

- **The four SSE job-slot hooks now share one implementation.** `useReferenceAudioImport`, `useYoutubeTrackImport`, `useVideoDownload`, and `useMidiTranscription` each hand-rolled the same "kick off a job → stream SSE progress → settle on the terminal frame, with double-click and lost-connection recovery" machinery. That logic now lives in exactly one place — a new generic `useSseJobSlot` hook — and the four feature hooks are thin wrappers that pass their own kickoff/SSE/cancel calls plus per-feature toast copy (and, for MuScriptor, its runtime-install and gated-repo modals). Behavior-preserving refactor deferred from the MuScriptor /simplify pass (#2368); ~150 fewer lines.

- **Shell page quick-command buttons trimmed to the interactive tools.** Dropped the four shell one-liner shortcuts (`git status`, `git pull`, `npm test`, `npm run dev`) that duplicated things you'd just type, renamed the `antigravity` button to its actual command `agy`, and added a `grok` button alongside the other coding-agent launchers.

- **Image and video generators now share one resolution control (#2436).** The image (`ImageGenControls`) and video (`VideoGen`) size pickers had diverged into two separate "preset dropdown + custom W×H" implementations over the same helper tables — and VideoGen rendered its Width/Height inputs twice. Both now consume a single `components/media/ResolutionField.jsx` (preset `<select>` + a "Custom…" sentinel that reveals gated Width/Height inputs with a blur-clamp), and `clampImageEdge` in `imageGenResolutions.js` is generalized to accept `{ min, max, step }` so video reuses it (clamping to 64–2048 and snapping to the multiple-of-64 grid the server floors to). The controls mirror their routes: the image edge inputs use `step=1` (its route accepts any integer edge, so a hand-typed 705 no longer trips the form's native step validation), while video keeps `step=64` and snaps on blur. Cleanup deferred from the custom-image-dimensions slice.

## Fixes

- **Generated agent prompts (swarm, scheduled claim-work, other queued system tasks) no longer double-print their first line.** A task queued through the CoS improvement/claim-work path is persisted to `COS-TASKS.md` by moving its full multi-line prompt into `metadata.context` and keeping only the first line as `description` (so the markdown stays one-line-per-task). Coming back out, `context` therefore led with a verbatim copy of `description`, and every prompt-render path emitted both — so a swarm run opened with the header twice (`# ⚡ SWARM MODE …`, then `### Context`, then `# ⚡ SWARM MODE …` again, the whole body repeated). A new `reconcileSplitContext` step in `agentPromptBuilder.js` detects that exact split signature (context's first non-empty line equals `description`), folds the body back into `description`, and drops the redundant `metadata.context` — so the prompt renders once as one clean body with no spurious `### Context` wrapper. A genuinely-separate user-supplied context (whose first line differs from the description) is left untouched, and the stored task keeps its one-line description for the task-list UI. Pure code fix — reaches every install on update, no data or prompt-template migration.

- **The CoS agent briefing no longer prints a redundant "Target Application" heading for the PortOS default app on the api-prompt path (#2430).** The light/fallback prompt path already suppressed the target-app line for `portos-default` (the agent runs in the PortOS directory, so it adds nothing), but the editable `cos-agent-briefing` stage template still rendered the heading for any task carrying an app id. The template now gates the stock heading on a precomputed `targetAppLabel` (empty for the default app, the app id for managed apps) that `agentPromptBuilder.js` threads into the render context — mirroring `buildTaskBlock`'s predicate — while leaving `task.metadata.app` in the context untouched for any custom template references. Ships as stage-prompt migration 187 (hash-matched, cross-synced with 010's accepted-hash chain) so unmodified installs auto-upgrade and customized copies are left alone.

- **Layered Intelligence's outcome feedback loop now reconciles the `plan` (PLAN.md) tracker too (#2435).** After #2428 closed the loop for forge (gh/glab) and jira trackers, the `plan` tracker stayed excluded because `readIssues` mapped every `[lil-*]` PLAN.md item to `state: 'open'` regardless of its `- [ ]` / `- [x]` checkbox, so a completed proposal could never resolve. `extractPlanSlugs` now preserves each item's checkbox state — a checked `- [x]` reads `closed` (which `deriveOutcome` maps to *merged*) and an unchecked `- [ ]` (or a bare tag with no checkbox — absent ≠ done) reads `open` — so a checked item both reconciles as an outcome and falls out of the dedup window to become re-proposable, while open items stay suppressed as before. The `outcomesTrackerSupported()` gate now includes `plan`, so PLAN-tracked apps learn from their own merge rate like forge/jira apps.

- **Character Sheet name edits can no longer be reverted by a stale blur-save that fires while the prior save is still in flight (#2409).** Committing a name change (Enter or blur) starts a PUT and immediately returns the header to its button view. If you reopened the editor before that PUT landed, the reopened input seeded from the pre-save name — and a subsequent blur could persist that stale value straight back over the change you just made. The name trigger now disables while a save is in flight (tracked via a `nameSaving` flag set before the PUT and cleared in `finally`) and `startNameEdit` refuses to reopen until the save settles, so exactly one write wins and no stale revert can slip through.

- **HuggingFace model downloads (Flux and every other image/video/music pull) now log to the server console.** Triggering a base-model download showed a progress bar in the UI but left no trace in the server log — the entire download path emitted progress *only* over the SSE stream to the browser, and PortOS has no HTTP access logger, so a headless/PM2 log had no record the multi-GB fetch ever ran. The shared SSE driver (`server/lib/sseDownload.js#startHfDownloadStream`, the single chokepoint behind the image, video, and music download endpoints) now writes single-line, emoji-prefixed log lines for the start of each repo pull (`⬇️`, with a forced-re-fetch marker), each file as it streams, a cache hit that skips the download (`📦`), the outcome (`✅` complete with byte count / `❌` failed with the error message), and client-disconnect cancellation (`🛑`) — while cancelled results are deliberately not logged as failures.

- **Audio → MIDI transcription now prompts for HuggingFace access instead of dead-ending on a gated-repo traceback.** MuScriptor's model weights turned out to live in a *gated* HuggingFace repo (`MuScriptor/muscriptor-*`), so the first transcription on a fresh install — even right after the runtime auto-installs — 403'd with a raw `GatedRepoError: Access to model … is restricted` traceback the user couldn't act on. The MuScriptor sidecar now runs under the shared `install_hf_error_handler` (the same cause-chain walker the image/video runners use), so a gated download emits a structured `USER_ERROR:gated_repo:<repo>` marker; the server classifies that into a typed `gated_repo` SSE frame (carrying the repo id), and the client opens the same license-accept + token-paste prompt the gated image models use (reusing `HfTokenBanner`): accept the model's license, paste a HuggingFace read token, and the transcription you originally asked for re-runs automatically. Gated-error detection is now shared via `isGatedRepoError`/`extractGatedRepo` in `server/lib/hfErrors.js` (also adopted by the image runner's prose fallback).

- **Layered Intelligence no longer silently wedges when pinned to an API-only provider.** After LI became an agent-backed scheduled task (#2322), its reasoning run needs a CLI/TUI coding provider (one with a file-writing harness to emit the `.agent-done` sentinel). Installs that carried an HTTP `api` provider (ollama / LM Studio) through migration 184 hit a dead end: every dispatch failed provider resolution *before* an agent registered, so nothing updated the task's status and it sat pending forever — silently blocking the app's whole improvement slot. Now the LI `buildTaskInput` hook detects an api-only provider up front and **skips** with an actionable reason (surfaced on the Intelligence tab: *"pick a CLI/TUI provider — e.g. the Claude Ollama sample"*), and, as a general safety net, any agent task that fails provider resolution with a **permanent** config error is **blocked** with a `provider-config` reason instead of re-failing on every dispatch.

## Features

- **CoS confidence auto-approval now has a safety-kind override, so outward-facing / irreversible work always needs human sign-off (#2440).** Confidence auto-approval used to gate on ONE axis — a task type's historical success rate — so once a type crossed the threshold, *any* task of that type could auto-run, including inherently outward-facing work: publishing content that goes live, federating records to sync peers, opening external/upstream PRs, or releasing/deploying. A new **safety-kind** axis, orthogonal to confidence, classifies each generated task (`server/services/taskLearning/safetyKind.js`): outward-facing/irreversible kinds (`content`, `publish`, `federation`, `external-pr`) always require the user's approval regardless of how high the success rate is — a 100% score can't undo work that has already left the install — while reversible internal work (analysis, refactor, code-review, same-repo improvement PRs) keeps today's success-rate gate unchanged. Classification is driven by explicit task metadata (`safetyKind` / `outwardFacing` / capability flags) then conservative keyword signatures, defaulting to *reversible* so existing behavior is preserved unless a task carries an outward signal. The reason is surfaced on the task record (`safetyKind` + `approvalReason`) and in the approval log line, so the UI can show why a high-confidence task still needs review. Tunable via the new `safetyKindApproval` CoS config (`{ enabled, alwaysApproveKinds }`, default on) — additive optional key merged on read, no migration needed.

- **The CoS Schedule Timeline can now expand each task track to see and edit its per-app overrides inline.** Which apps run a given scheduled task type (and each app's interval / agent-option / issue-author / swarm overrides) was previously only visible on the Schedule tab's per-task drawer — the Timeline showed the task cadence but not who it runs for. Every task track — both the pinned timeline rows and the unpinned rotation/on-demand tasks in the "Unpinned runner queue" — now carries a chevron that expands to the same `PerAppOverrideList` used on the Schedule tab (enable/disable per app, "Enable/Disable All", per-app interval and metadata), plus an `enabled/total` app-count badge (shown even at `0/N` so "runs for none of my apps" is distinguishable from "no app scope") so an at-a-glance "3/5 apps" is visible without expanding. Edits save through the existing per-app override endpoints and refresh the timeline in place. System-job tracks (which are not per-app) show no expand toggle. The timeline graph node now forwards the `appOverrides`/`enabledAppCount`/`totalAppCount` plus the global `taskMetadata`/`managedAgentOptions` defaults that `getScheduleStatus` already computes; no migration needed (read-only surfacing of existing data).

- **Layered Intelligence now evaluates a managed app against its OWN goals and performance — not PortOS's agent telemetry.** Running the loop on a managed app previously fed it the PortOS install's global autonomous-agent run stats (`data/cos/learning.json`) as the "CoS metrics" source — which is on by default for every app — so a managed-app run reasoned over PortOS's tooling performance instead of the app's own product performance. Two changes fix this: (1) a new **App metrics (METRICS.md)** telemetry source, on by default for every app, reads a repo-root `METRICS.md` from the app (same as `GOALS.md`/`PLAN.md`) — the place an app documents how it measures success (user-success KPIs, production telemetry, where its data lives, and read-only queries an agent can run); and (2) the **CoS metrics** source now defaults **on only for the PortOS install** (where the agent stats genuinely are its own-performance metric) and **off for managed apps** (where they're a tooling/interaction signal, still opt-in). The reasoning prompt is reframed around "how is THIS app performing against its own goals and purpose," and when a managed app exposes no own-performance metrics the loop is nudged to propose adding a `METRICS.md` as an `app-data-gap`. New standard documented in [`docs/METRICS.md`](../docs/METRICS.md); apps can also wire a custom `cmd`/`http` source to feed live numbers into each run. Back-compat: these are default changes only — any explicitly-stored source toggle is preserved, so no migration is needed.

- **Grok is now selectable as a Review Loop code reviewer.** The reviewer picker (AI Providers → Code Review Defaults, plus the per-task task form and the per-task-type Schedule config) gains a **Grok** option alongside Claude, Codex, and Antigravity. Selecting it adds `grok` to slashdo's `--review-with` list, and the review-loop follow-up agent invokes the Grok Build CLI (`grok`) headless against the PR diff — the same CLI-reviewer path the other coding-agent reviewers use (it already routed any non-Copilot/non-local-LLM reviewer generically). New enum value `grok` in `REVIEWER_VALUES` (mirrored in the client `REVIEWER_OPTIONS`); no migration needed (absent from a stored list = not selected, and the default reviewer stays `copilot`).

- **Code Review Defaults can now gate merges on arbitrary GitHub reviewer usernames (e.g. `@CodeReviewbot`).** The Review Loop was limited to a fixed set of reviewer kinds (Copilot, the Claude/Antigravity/Codex CLIs, and the local LM Studio/Ollama backends); there was no way to require a specific GitHub bot or teammate as a PR reviewer. The reviewer picker (AI Providers → Code Review Defaults, plus per-task in the task form and per-task-type in the Schedule config) grows a **"GitHub reviewers (gate merge)"** row where you type any GitHub username or `org/team` slug. Those usernames are appended to slashdo's `--review-with` as `@user` tokens after the keyed reviewers, and the review-loop follow-up agent is instructed to request each as a PR reviewer (`gh pr edit … --add-reviewer`), wait for their review, and treat their approval as a merge gate — end to end through the original-agent completion prompt, the follow-up loop, and the `/do:next` claim flow. Usernames are stored without the `@`, validated against a shell-safe GitHub-username charset (1–39 chars, alphanumeric/hyphen, optional `/team`), case-insensitively deduped, and capped at 20, so a hand-edited `settings.json` can't smuggle in an unsafe or oversized token. They're a distinct class from the behavior reviewers — external PR reviewers that gate the merge, not CLIs that apply fixes — so `--reviewer-applies` never turns on for a username-only addition, and a username reviewer is forge-agnostic (kept even when Copilot is stripped on a non-GitHub forge, and able to drive the loop on its own). New config key: `settings.codeReview.usernames` (+ per-task `metadata.usernames`); no migration needed (absent = none).

- **Image Gen now supports arbitrary (larger) output dimensions, not just the fixed preset list.** The resolution dropdown gained a **Custom…** entry that reveals Width and Height number inputs, so sizes the presets never covered — e.g. 704×1280 (9:16 portrait) for Z-Image — no longer require adding a new preset. Any edge in 64–3840px is accepted as long as the total stays ≤ 8,294,400px (the same `imageEdgeSchema` + pixel cap the server already enforced); a below-minimum edge snaps up to 64 (on blur, and again at submit so an Enter-key submit can't slip a sub-64 value through), and an over-cap total shows an inline warning before you submit. Custom mode also auto-engages whenever the current size matches no preset (a remix, an uploaded init image, or a stale off-backend size), so the exact dimensions stay visible and editable instead of collapsing to a read-only "(custom)" label. The inputs live in the shared `ImageGenControls`, so the Image Gen page, Universe Builder batch render, and the pipeline image-gen drawers all get custom dimensions at once.

- **Layered Intelligence now learns from its own results — the feedback loop is closed (#2428).** The LI reasoner used to file improvement proposals as tracker issues and then never read back whether they were merged, rejected, or abandoned, so it proposed in a vacuum (re-suggesting scopes with a 0% adoption rate). LI now records every filed proposal in a lightweight per-app outcome store (`data/cos/li-outcomes/`, a `createCollectionStore` collection with per-slug records and 30-day auto-GC), reconciles each proposal's outcome from the live tracker state on the next run (GitHub `stateReason` maps *not-planned* → rejected, any other close → merged; still-open stays unresolved), and folds a merge-rate report — totals, per-scope adoption, common rejection reasons — back into the reasoning prompt with guidance to prefer higher-merge scopes and justify low-merge ones. Gated behind a new `Proposal outcomes` telemetry source toggle in the per-app LI config (default **on** for the PortOS install, **off** for managed apps). No migration needed — the store is created on first use and existing LI behavior is unchanged when the toggle is off.

- **Audio → MIDI transcription now lets you pick the MuScriptor model size (small / medium / large) per request.** The transcribe controls always ran the `medium` weights because the client never sent a size, even though the server route, Zod schema, and sidecar already accepted an optional `model`. Both trigger points — *Transcribe MIDI* on a Rounds reference audio and *MIDI* on a Music Video source track — now show a size selector next to the button. Larger tiers give higher-quality transcriptions but are slower and pull a bigger weight file on first use; `medium` stays the default so existing behavior is unchanged, and the chosen size is threaded straight into the existing `transcribe-midi` endpoints. The selector disables while a transcription is in flight.

- **Audio → MIDI transcription now installs its runtime automatically on first use.** Clicking *Transcribe MIDI* (Rounds reference audio) or *MIDI* (Music Video source track) used to dead-end on a toast telling you to run `INSTALL_MUSCRIPTOR=1 bash scripts/setup-image-video.sh` in a terminal. Now, when the MuScriptor venv isn't provisioned yet, PortOS opens an in-app installer that streams the setup line-by-line — the same experience as the image and video model runtimes — and, on completion, automatically runs the transcription you originally asked for. A new `GET /api/midi-runtime/install` endpoint reuses the shared `RuntimeInstallModal`/SSE-install path (in-flight singleton, detached process group so a cancel kills pip/git children, post-install venv re-probe); `useMidiTranscription` catches the `503 MIDI_RUNTIME_MISSING` kickoff, gates the modal, and retries once installed (with a loop guard so a genuinely-failed install surfaces its error instead of relooping).

- **PortOS now installs a service worker for offline app-shell loading and low-bandwidth caching** — the modern replacement for the long-deprecated AppCache manifest. After the first online visit, the built app shell (`index.html`) and the content-hashed JS/CSS chunks are cached, so the UI loads instantly on repeat visits, keeps rendering when the server is briefly unreachable (flaky Tailnet links), and stops re-pulling megabytes of vendor JS on every navigation over a slow connection. Google Fonts and static public assets (icons, textures, manifest) are cached too. Real-time and large/dynamic routes — `/api/*`, `/socket.io/*`, and `/data/*` media — are **never** cached, so live data and streamed assets always hit the network (HTTP Range preserved for `/data`). Navigations stay network-first so a fresh build is always picked up when online, and the existing build-id / stale-chunk-reload machinery still handles "the server rebuilt while a tab was open." The worker registers only in a production **secure context** (HTTPS, or `localhost`) — it is a progressive enhancement that silently no-ops over a plain-HTTP Tailnet hostname and in the Vite dev server (where it also tears down any stale worker). Implemented as a hand-rolled worker (`client/public/sw.js`) with no new dependency; registration lives in `client/src/lib/registerServiceWorker.js`. The PWA manifest gained explicit `id`/`scope` for a deterministic installed-app identity.
