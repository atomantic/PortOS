# Unreleased

## CoS self-learning reliability

- **[issue-2620] Layered Intelligence stops re-proposing finished work** — completed plan items stay recognized as done, and proposal outcomes now distinguish merged, rejected, and abandoned instead of counting every closed issue as merged.
- **[issue-2617] Task learning forgives fixed problems** — scheduling, model-routing, and self-diagnostics decisions now weigh recent runs instead of lifetime totals, so a task type recovers as soon as it starts succeeding again.
- **[issue-2619] Self-heal cleanup after a fixed failure storm** — a health-tick reaper now auto-expires stale failure-blocked tasks (older than 14 days) and investigation tasks whose origin is gone/completed via a federation-safe status flip (never deletion); resetting a task type's learning now also purges its stale failure-signature samples and correlation-window rows; failure signatures are age-windowed (30 days) so an outage-era sample stops steering routing; and a poor-lifetime task type whose recent window is clean is now auto-rehabilitated even while it keeps running (not just when idle).
- **[issue-2615] One investigation per failure cause, not one per failure** — repeated agent failures now dedupe into a single investigation task, with an hourly cap and no investigations-of-investigations.
- **[issue-2618] Provider outages no longer count against task learning** — rate-limit, auth, billing, and startup failures are tracked separately instead of denting a task type's success rate, so an outage can't get a healthy task skipped or demoted.
- **[issue-2621] Feature brainstorming remembers rejected ideas** — the feature-ideas task now checks REJECTED.md and previously closed-unmerged proposals before suggesting, so declined ideas stop coming back.
- **[issue-2614] Failing scheduled tasks no longer pile up duplicates** — a task blocked by repeated failures now counts as "already queued," so the same improvement task isn't re-created every cycle. Explicit retries (an on-demand Run, a manual job trigger, a Creative Director re-generate, or a repeated voice dispatch) revive the blocked task with a fresh retry budget instead of silently doing nothing, and the CoS log names the blocking task whenever it suppresses new work.

## Fixed

- **Deleting a blocked CoS task now clears the "N blocked tasks" alert bar immediately.** On the Chief of Staff → Tasks page, the `ActionableInsightsBanner` (which owns its own `/cos/actionable-insights` fetch, decoupled from the task list) kept showing a stale blocked-tasks alert after a blocked task was deleted, until its 60s poll fired or the user manually dismissed it. The page now bumps an `insightsRefreshKey` on every task-list mutation, which the banner watches (via a mount-skipping effect) to `refetch()` its server-derived insights — so a delete (or any task mutation through the tasks list) re-derives the banner right away. Follow-up #2654 tracks lifting the insights fetch into the page's `fetchData` to also cover socket/health-driven staleness.

- **claim-issue no longer reports "no open issues" when an author filter is hiding a full queue.** The perpetual `claim-issue` / `claim-issue-gitlab` work-detector runs `gh`/`glab issue list` with an author filter (`self`→`@me`, or `owner`→the repo owner), and an empty *filtered* result was flattened to the reason `no-open-issues` — so a repo with open issues that simply don't match the filter parked for a full recheck cadence claiming there was nothing to do. The two failure modes this hid: `owner` resolves to an **org** (e.g. a GHE repo owned by a team), which is never an issue author, so the filter matches nothing; and `self`/`@me` is the wrong identity when the forge session differs from whoever filed the issues. The detector now re-probes without the author filter when the filtered list is empty, and if open issues exist it parks with a new `no-authored-issues` reason carrying the real open count. The on-demand "Run" toast now says *"open issues exist, but none match the author filter — set it to \"any\" (0 of N open)"* instead of the misleading "no open issues", pointing the user at the fix. A genuinely empty repo still reports `no-open-issues`, and the `any` filter is unaffected (no wasted re-probe). New private helper `countOpenIssuesUnfiltered`; covered by `perpetualWork.test.js` and `useOnDemandTaskToast.test.jsx`.
- **claim-issue distinguishes the "owner filter matches an org" trap from a personal-username mismatch.** Building on the fix above: when the `owner` author filter resolves to an **organization** login (the common case for team/enterprise repos), `--author <org>` can only ever match zero issues — an org is never an issue author. The detector now recognizes an org owner up front (`gh repo view --json owner,isInOrganization`), skips the guaranteed-empty query entirely, and parks with a distinct `owner-is-org` reason carrying the real open count. The toast now reads *"the \"owner\" filter matches an org, which can't author issues — set it to \"self\" or \"any\" (0 of N open)"* instead of the generic "set it to any", so the user isn't left thinking their own username is mismatched. `owner` mode on a user-owned repo is unchanged. Covered by `perpetualWork.test.js` and `useOnDemandTaskToast.test.jsx`.
- **pr-watcher now works on GitHub Enterprise hosts and resolves "self" against the repo's actual host.** The `pr-watcher` task gated on `getOriginInfo().isGithub`, which is github.com-only, so it silently no-op'd (`not-a-github-repo`) on self-hosted GitHub Enterprise (`github.*`) repos. Worse, its self/others author gate resolved the operator login via `gh api user` with no `--hostname` — which always targets github.com regardless of the repo — and cached one login process-wide, so on an enterprise repo it compared PR authors against the wrong identity (a user is commonly a different login per host). The gate now classifies GitHub-family hosts via the shared `hostToWorkTracker` (the same rule the claim-issue router uses; `gitlab.*` still falls through as not-a-github-repo), the default-branch and PR-list `gh` calls pass a host-qualified `--repo HOST/OWNER/REPO` selector (enterprise-correct, and deterministic on a fork+upstream multi-remote checkout where cwd-based detection would ambiguously resolve to the parent), and `getSelfLogin(host)` resolves identity with an explicit `--hostname` and caches per host. Covered by `prWatcher.test.js`.
- **Retry lazy route chunks before failing on flaky mobile networks.** Lazy-loaded pages fetch their JS chunk on demand, and on a weak cellular/Wi-Fi link that fetch can fail on a transient blip ("Importing a module script failed" on Safari, "Failed to fetch dynamically imported module" on Chrome) even though the chunk still exists — a page reload can't fix a network hiccup, so once the one-shot stale-chunk reload was spent for a build, the next blip stranded the user on the error screen. `lazyWithReload` now retries the dynamic import twice with a short backoff before treating the failure as terminal, so a transient mobile-network failure recovers silently without a jarring reload; a genuinely stale chunk (renamed by a rebuild) still 404s on each quick retry and falls through to the existing reload-once path. New `importWithRetry` helper with unit coverage.

- **Clear the offline (service-worker) cache before the stale-chunk recovery reload.** When a genuinely stale chunk (renamed by a rebuild) triggered the one-shot reload, the reload could be served the *old* bundle right back — the service worker serves `/assets/*` cache-first, and on a flaky mobile link its network-first navigation falls back to the cached (old) app shell. That re-loaded the very code that just 404'd and burned the build-scoped reload guard, so the user stayed stuck on "Importing a module script failed" even after the reload and even on the latest build. `reloadOnceForStaleChunk` now purges the SW's `portos-*` caches (app shell + hashed asset chunks) via the page-side Cache Storage API before reloading, so the recovery reload is guaranteed to boot fresh code; the purge is bounded by a 1.5s timeout so a hung/disabled Cache Storage still reloads. The purge is gated on a build-id check: a flaky/offline network produces the same import-error messages as a stale chunk, and purging on a transient failure would destroy the valid current-build offline shell (if the link drops again during the recovery reload, the SW has no fallback and the user lands on the browser's connection-error page). `fetchServerBuildId` re-fetches the shell with `cache: 'no-store'` and reads the `portos-build-id` meta the server stamps into it — the caches are dropped only when the server confirms a *different* build; unreachable, non-ok, same-build, and unstamped responses all keep the caches so the reload can boot the SW's cached shell. New `purgeOfflineCaches` + `fetchServerBuildId` helpers with unit coverage (`staleChunkReload.test.js`).

## Security

- **[issue-2514] Harden AI provider custom endpoints against SSRF / API-key exfiltration.** A provider's `endpoint` is a free-form URL, and model-refresh / API-run paths attached the paid `Authorization: Bearer <apiKey>` and fetched that host with no host checks — a hostile or mistyped endpoint could receive the user's paid LLM key and prompts, or be pointed at a cloud-metadata service (`169.254.169.254`). A new `endpointGuard` now gates every secret-bearing request: loopback/LAN (Tailscale) and known paid-provider hosts are allowed, cloud-metadata hosts are always blocked, and any other host requires the new per-provider **Allow custom endpoint** opt-in. Keyless local-LLM calls (Ollama/LM Studio) are unaffected.
- **[issue-2518] Harden upload path containment and stop leaking absolute FS paths.** The upload/attachment/screenshot routes guarded writes with a bare `resolvedPath.startsWith(DIR)` (no trailing separator, so a sibling like `data/uploads-evil/` string-matched) and the attachment/screenshot responses echoed the absolute on-disk `path` back to the client, leaking the install layout / home-directory username segment. Containment now goes through a shared `isPathInsideDir()` helper (anchored `resolvePath(dir) + sep` prefix, matching `makePathResolver`), responses return only API-relative URLs (`/api/screenshots/<file>`), and the CoS prompt builder resolves those refs back to absolute paths at render time (legacy absolute values pass through unchanged). The `moltbook-claim.mjs` script now logs only the token length + short prefix instead of the full bearer token.
- **[issue-2517] Revalidate the browser navigate URL by resolved IP and by Chrome's actual per-hop connect address.** `POST /api/browser/navigate` only checked the host LITERAL with `isSafeIngestUrl`, so a hostname whose A record rebinds to loopback/link-local/cloud-metadata, or a post-navigate redirect the CDP browser silently follows to a blocked host, slipped past. The route now adds a DNS-resolve pre-check (`assertPublicHttpUrl`) that refuses a hostname resolving to a blocked address before any tab is opened, and drives the navigation through `navigateToUrlPinned` — the same connect-time pin the catalog URL ingest uses — which verifies Chrome's own reported `remoteIPAddress` for the initial nav, every HTTP redirect, and any client-side navigation in the settle window, tearing the tab down and failing closed on a blocked hop. LAN/Tailscale hosts stay intentionally allowed. Residual: a WebSocket to a hostname that DNS-rebinds to a private IP can't be pinned via CDP interception (no `remoteIPAddress` on WS events) — bounded by the single-user trust model and documented in `browserService.ssrfPinRefusalReason`.
- **[issue-2515] Restrict Layered Intelligence custom `cmd` sources to allowlisted binaries (no `shell: true`).** A custom `cmd` source ran its full command string through `shell: true` with only a length/timeout cap, so any config-write path (hand edit, hostile sync payload) gained persistent, unattended RCE on the LI schedule (`; rm -rf ~`, `$(curl … | sh)`, pipes to `sh`). `runShellCommand` now denies the shell by default: the command is parsed and checked against the shared `commandSecurity` binary allowlist (rejecting shell metacharacters and non-allowlisted binaries), then spawned with `shell: false` and parsed args.
