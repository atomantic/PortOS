# Unreleased

## 3D

- **[issue-2952] Turn a rendered image into a downloadable 3D mesh (create/render backend)** — once TRELLIS.2 is installed, PortOS can now take one of your gallery images and render it into a real 3D model (a `.glb` mesh with PBR textures) entirely on-device. Each render is saved as its own record you can list, re-render, download, or delete, and it tracks live status (generating → ready, with progress) and its render history — the backend "create a 3D model" path the 3D page's UI drives. The render only ever runs when you ask for it (never in the background), and is refused with a clear reason if the model isn't installed or your hardware can't run it.
- **[issue-2953] Install image-to-3D models from a new 3D page** — a new **3D** page under Create lists the available image-to-3D models (starting with Microsoft TRELLIS.2) and shows each one's state at a glance: **Ready**, an **Install** button, or a note explaining why your hardware can't run it (e.g. needs an Apple Silicon Mac with enough memory). Installing runs entirely on-device with a live progress log — the same in-app install experience as the image generators — so you never touch a terminal. Reachable from the sidebar, ⌘K, and voice. (Turning one of your rendered images into a 3D mesh from this page lands next.)
- **[issue-2952] TRELLIS.2 install now survives network hiccups** — the multi-gigabyte TRELLIS.2 install pulls from several code repositories, and a single dropped connection mid-download used to abort the whole thing with a cryptic error. The installer now automatically retries a step that fails from a transient network drop (with a short backoff), and already-downloaded pieces are kept — so a flaky connection resumes instead of starting over. If it still can't finish, the error now says plainly that it was a network hiccup and that clicking **Install** again picks up where it left off.

## Internal

- **[issue-2958] Polyfill jsdom `scrollIntoView` in the client test setup** — jsdom doesn't implement `Element.prototype.scrollIntoView`, so a `requestAnimationFrame` callback that calls it (e.g. `Tribe.jsx`, `CmdKSearch.jsx`, and other components) could fire after its test unmounted and turn into an unhandled error that failed the *entire* client suite despite every assertion passing — an intermittent CI flake that forced no-op re-runs on unrelated PRs. `client/src/test/setup.js` now installs a guarded no-op `scrollIntoView` stub alongside the storage polyfill, killing the whole class of flake in one place. Production is unaffected (the stub only fills the jsdom gap).
- **[issue-2951] Image-to-3D target registry (foundation)** — added a pluggable `server/services/imageTo3d/targets.js` registry (mirroring `imageGen/modes.js`) that models image→3D models as selectable targets with hardware-gated availability resolution, TRELLIS.2 registered as the first target, and a read-only `GET /api/image-to-3d/targets` endpoint. Backend groundwork for the forthcoming 3D page under Create (#2952 install/runner, #2953 page/viewer, #2954 CUDA lane); no user-visible surface yet.
- **[issue-2952] TRELLIS.2 local runner scaffolding** — added `server/services/imageTo3d/trellis2.js`: on-disk install detection, pure install/generate command builders, a format-agnostic `generate.py` progress parser, and a guarded generate runner (throws unless installed; never auto-runs). `GET /api/image-to-3d/targets` now also reports each target's `installed` state. The real ~15 GB install + live GPU render are user-triggered and validated hands-on (remainder of #2952); no user-visible surface yet.
- **[issue-2952] TRELLIS.2 SSE install endpoint** — added `GET /api/image-to-3d/trellis2/install` and `installTrellis2()`, which clone the Apple Silicon MPS port and run its `setup.sh` while streaming staged progress (stage → log → complete/error). Gated on hardware support (refuses on non-Apple-Silicon / insufficient memory), single-flighted against double-clicks, and canceled if the client disconnects. The real ~15 GB download only fires on this explicit user request (never at boot); the orchestration is mock-tested. No user-visible surface yet (the install button lands with the 3D page, #2953).

## Sprite Manager

- **Steer a directional-anchor re-roll with a correction note** — when a directional anchor comes out wrong (say the north-east pose grew a pocket on the arm that has none in every other view), re-rolling it with the same inputs tended to just reproduce the same mistake. Each unlocked anchor tile now has a small **Correction (optional)** box: type what to fix ("no pocket on the right sleeve") and hit **Regenerate**, and the note is appended to that pose's prompt as an explicit override so the new render actually diverges. The button reads **Generate** on the first render and **Regenerate** once candidates exist, the correction is per-direction (so eight different fixes don't collide), and it rides into the candidate's saved prompt so the image preview shows exactly what was asked for. Main-reference iteration is unchanged — it already has its own design-prompt box.
- **Locking a main reference no longer leaves a colored halo around the character** — when you lock a character's main reference, PortOS re-keys the background to whichever chroma color (magenta, green, or blue) sits farthest from the character's own palette. If that swapped the background — e.g. a magenta render re-keyed onto blue — the anti-aliased pixels along the character's outline kept their old magenta tint and showed up as a faint magenta ring against the new blue backdrop. Locking now cleans those edge pixels as it re-keys, so the outline blends into the new background with no leftover halo. Already-locked references (which are immutable and can't be re-locked) are healed automatically on upgrade: each one is re-derived from its saved candidate with the fix applied, and its stored checksum is refreshed — no action needed on your part.
- **Generate Walk is faster, more reliable, and you can watch grok do it** — the walk-cycle generator got three fixes and a new window into the render:
  - **Watch grok in the Shell.** Clicking **Generate walk** now runs grok as a live, interactive TUI session instead of a headless background job. The direction's card shows a **Watch in Shell** link while it renders — click it to pop into the Shell page, watch grok animate the frames in real time, and (if it goes off the rails) type to course-correct or Stop it. The session goes away once the clip is rendered and the deterministic packaging takes over.
  - **Shorter clips.** Walk clip length is now selectable from **1 / 2 / 3 / 6 / 10 seconds and defaults to 2s** (was 6s/10s only). A looping walk cycle only needs one gait (~8 frames), so a 2s clip renders in a fraction of the time — the old 6s default took ~10 minutes to render a clip we harvest 8 frames from.
  - **Fixed the "not a usable matte" failure.** Grok clips that faded in from a black frame, or rendered a slightly off-magenta background, used to fail packaging outright with `Measured background [0,0,0] is not a usable #FF00FF matte` — throwing away a perfectly good clip. The postprocess now drops black lead-in/out frames and tolerates grok's real-world magenta. The deeper cause is also fixed: we now hand grok the character **on its magenta matte** (the anchor already has it) instead of a transparent cut-out it had to composite over black and re-invent a background for — so the rendered clip keys out cleanly.
  - **A failed render isn't a dead end.** If packaging still fails, the card now plays grok's raw clip inline (alongside the error and a Retry) so you can see exactly what grok produced instead of staring at an error string.

- **See — and copy — the prompt behind any sprite image** — click a reference-set image (the frozen main, a locked directional anchor, or any generation candidate) and the enlarged preview now shows the exact prompt that produced it, with a one-click **Copy prompt** button, the same way the image render-history preview does. The asset inspector gains the same prompt block, so a walk-animation clip or any generated asset you open there shows its prompt too. Older images generated before this shipped still show their prompt — it's reconstructed from the render's recorded design description, direction, and background color — so nothing in your existing library is blank. Imported artwork and internal metadata files simply omit the block. New endpoint `GET /api/sprites/:id/asset-prompt`.
- **Seed a character's main reference from an existing sprite, an upload, or your render history — and fork a character from its reference** — the "design image (optional)" upload on the main-reference form is now a three-way **Reference image** picker: upload a file, pick one from your image render-history gallery, or pick another sprite that already has a locked reference, each with a fidelity slider controlling how tightly the render follows it (image+text→image). Separately, a locked reference is immutable and can never be regenerated — so a character with a frozen main now shows a **Fork from this reference** button that spins up a brand-new character whose main is generated from this one's reference plus a prompt describing the change ("same character but in a red coat"), then opens the new sprite. New endpoints `GET /api/sprites/reference-sources` and `POST /api/sprites/:id/fork`.
- **The Sprite Manager opens on a full Library catalog, with rename and delete built in** — landing on the Sprite Manager now shows the **Library** itself: a searchable, grouped grid of every character/place/object, each card carrying a real preview thumbnail — a character's locked main reference, or, for a place/object (or a character with no reference yet), the first image asset on disk, falling back to a kind icon only when there's no image at all — instead of jumping you straight into one sprite. The catalog is the `/sprites` route (not a modal buried behind a button), so it's directly linkable, and each card gives you full record control: click to open it, the pencil to **rename** in place (for records whose generated name you want to fix), or the trash to **delete** behind a cautious inline Cancel/Delete confirm (a delete drops the record from your library; its files stay on disk and its id stays reserved). The same **rename** and **delete** controls sit in a sprite's detail header, so you can fix or remove the one you're looking at without going back to the grid (deleting it returns you there). Search, **New Sprite**, and **Import** live in the page header (New Sprite/Import are modals), the left sidebar is gone so the catalog and detail panes run full width, and opening a sprite shows a **Library** link back to the grid. The per-character workspace tab that used to read "Library" is now **Assets**, since "Library" means the catalog. New endpoint `GET /api/sprites/thumbnails`.
- **Click any reference-set image to enlarge it** — the Reference Set's inline thumbnails (the frozen main reference, each locked directional anchor, and every generation candidate) are now click-to-zoom: clicking one opens a large, pixel-perfect preview over the same checkerboard the asset inspector uses, so you can actually judge a candidate before locking it instead of squinting at a thumbnail. Previews elsewhere are unaffected unless they opt in.
- **Atlas manifests are folded into the atlas, not a confusing "Manifests" shelf** — a character's asset library used to show a separate **Manifests** group of near-identical JSON cards sitting apart from the sprite-sheet PNGs they describe, so it was never clear that each manifest is just the build metadata for one atlas version. Manifests are no longer their own group: a runtime atlas card gains a **Manifest** button that opens its sidecar (the geometry, chroma key, and source provenance the compiler recorded), and the inspector now says plainly that it's behind-the-scenes metadata removed together with its atlas. The other internal JSON (publish pointers, run records, the reference-set manifest) no longer appears as inert cards at all. Nothing changes on disk — manifests are still written and still deleted as a unit with their atlas version.
- **Every walk direction is trimmable now, whatever produced it** — a direction whose art came from the source pipeline's older video-first "redraw" path (pioneer's *east* being the case in the wild) showed its "approved ✓" card with no **Edit in Loop Trimmer** link, because loop trimming was wired only to grok-generated runs. The Loop Trimmer now resolves any run's geometry from the run's own manifest/strip regardless of where it was authored, so imported and redraw directions are trimmable exactly like native ones — the link appears on every direction that has a packed strip. (Saved trims stay preview-only; there's no run behind them to re-trim.)
- **Sprite runs are stored vendor-neutrally** — under the hood, PortOS used to file its own walk generations under a `grok/` directory, baking one provider into the on-disk layout. Runs now live under a neutral `runs/` tree with the provider recorded as run metadata instead, so future agent services or generation methods slot in without a storage change and nothing downstream (trimming, atlas compile, publish, backups) assumes a vendor. Existing installs are migrated automatically on upgrade: each character's `grok/<run>/` directories are renamed to `runs/<run>/`, every embedded path is rewritten, and the checksums that pin the finalized walk set + compiled atlas are recomputed so a re-compile stays a no-op — your published atlases are unaffected. Imported/redraw characters and the atlas image bytes are left untouched.
- **Delete old atlas and reference images from the asset inspector** — as you iterate a character, its `runtime/` tree fills up with superseded atlas versions (`…-animation-atlas-v10.png`, v9, v8…) and its reference/walk trees fill up with old candidate renders, with no way to prune them short of hand-editing `data/sprites/`. The asset inspector now has a **Delete** action: open any asset, click Delete, confirm in an inline Cancel/Delete row, and it's removed from disk (the library refreshes in place). Deleting an old runtime atlas removes its sidecar manifest in the same step, so a version goes cleanly. The one atlas you can't delete is the current published one — PortOS refuses it (and the pointer/publish-history bookkeeping) so you can't strand a live game or the recompiler; compile or publish a newer version first.
- **Unlock a finalized walk set to revise it** — a frozen 8-direction walk set was one-way: the "walk set frozen · immutable" badge was the end of the road, and the only way to change a direction was to start a whole new character version. The Walk Cycles panel now offers an **Unlock** button beside that badge (with an inline "Re-open all directions?" confirm). Unlocking removes the frozen set and re-opens every direction back into the generate / regenerate / approve flow — your already-rendered clips are kept on disk, so you can re-approve the ones you like and re-render just the ones you don't; re-approving all eight re-freezes the set exactly as before. Walk sets imported from an external source pipeline (which have no regenerable clips behind them) still can't be unlocked and say so.
- **The Loop Trimmer auto-plays when you open its tab** — switching to the Loop Trimmer now starts the loop preview animating immediately instead of waiting for a manual Play click; switching the animation source keeps it playing.
- **The asset inspector is readable again** — clicking an atlas or any sprite asset opens the inspector over a solid card and a darker backdrop, so the metadata (path, dimensions, frame count, size) and the Download / Open / Copy-path buttons no longer have the page bleeding through behind them.
- **The Directional anchors grid collapses once complete** — while you're building the reference set the "Directional anchors" grid stays open so you can generate and lock each direction, but once all eight are locked it's just static previews of files the "Reference set" browser below already lists (and lets you inspect / download). It now collapses by default when complete — with a one-click toggle to reopen it — so the same anchors no longer read as a duplicate set on the page.
- **[issue-2933] Trim walk loops in a real editor, not a buried 8-checkbox strip** — loop trimming used to be a tiny row of checkboxes tucked inside a walk candidate card, with no way to see what you were doing: no playback, no scrubbing, no frame rate control, no per-frame preview. It's now its own **Loop Trimmer** workspace on the Sprites page, switchable from a tab beside the asset **Library** and deep-linkable (`?spriteTab=trimmer&run=<id>`) so a specific run is shareable and bookmarkable — and "Edit in Loop Trimmer", from either a walk card or an asset tile, jumps straight to it. Pick an animation source (any packaged walk run, plus your previously saved trims for review), play the loop on a checkerboarded canvas or scrub it frame by frame with prev/next and a frame label, tune the frame rate with a live slider, and toggle individual frames on and off from a grid of per-frame thumbnails (disabled frames dim and desaturate, the current frame is outlined) with a live enabled-count and Enable-all / Invert shortcuts. Name the output (it's normalized to a safe `a-z0-9-` slug) and save — the trimmed strip, preview GIF, and manifest paths are surfaced right there, and the source artwork is never touched. The old inline trim panel is gone, so there's exactly one place to trim.
- **[issue-2932] Search the sprite library, and organize it by character / place / object** — the Sprites sidebar used to dump every record as a flat wall of names under just two buckets ("Characters" and "Props"), so a standalone place or object had nowhere to live and a big library was hard to scan. It now leads with a labelled autocomplete search box: type any part of a name, id, or kind and it narrows to a bounded, keyboard-navigable suggestion list (↑/↓ to move, Enter to open, Esc to clear); the full list is one click away behind a **Browse all** disclosure. Records group under **Characters**, **Places**, and **Objects**, and the New button (now **New Sprite**) lets you pick which of those three you're creating. Existing imported prop-atlas families keep working unchanged — they simply show under Objects alongside standalone objects. The reference / walk / publish workflows remain character-only.
- **[issue-2938] Asset badges and Regenerate now reflect approval and backend state, not just the file path** — a walk direction you've already approved now shows its strip, frames, and animation with an **approved** badge instead of "candidate": approval is tracked in the walk selection, and the artwork never moves on disk, so the browser now reads that selection to badge the assets correctly. And a reference candidate's **Regenerate** button is greyed out with the same "No image backend configured" guidance the reference workflow shows when no image backend is set up — so it no longer queues a render that just fails with a toast — and when you do re-roll a candidate, it uses the same backend you picked in the reference workflow above instead of falling back to the install default.
- **[issue-2931] A character's assets now group by what they are, and you can act on them in place** — the asset browser used to be a flat dump grouped by folder name (`reference`, `grok`, `runtime`), with every tile inert. Assets are now grouped by their actual role — reference set, strips, animations, frames, atlases, review evidence, manifests — and each one carries a status badge (runtime · approved · candidate · superseded · rejected · source), so you can see at a glance which reference version is current and which have been replaced. Walk strips, frames and animations gain two buttons: **Regenerate**, which re-runs that direction's render, and **Edit in Loop Trimmer**, which jumps straight to the trimmer for that run — including for a direction you've already approved, since trims are saved separately and never overwrite the approved artwork. Regenerate is correctly greyed out when the walk set is finalized, when the direction is already approved, when its reference anchor isn't locked yet, or while a render for it is already running — and it now shares one in-flight view with the workflow panels above, so starting a render in one place immediately disables it in the other instead of letting you pay for the same render twice. An unlocked reference candidate can be re-rolled the same way. Switching between characters no longer carries a stale "Rendering…" state across from the previous one.
- **[issue-2930] Sprites is now its own page, and assets are properly inspectable** — the Sprite Manager was a hidden tab inside Media Gen; it now has its own **Sprites** entry in the Create sidebar and lives at its own address, so it's directly linkable, bookmarkable, and reachable from ⌘K and voice. Old Media Gen sprite links (including links to a specific sprite) redirect to the new page and keep any deep-link state. Every surface that shows a transparent sprite — asset thumbnails, the reference set, walk-loop previews, trim results, and the compiled atlas — now paints a checkerboard behind the artwork, so a cut-out region is finally distinguishable from a black pixel instead of both reading as the same dark rectangle. Clicking any asset opens a real inspector: a pixel-perfect preview over that checkerboard, plus the file's path, dimensions, format, frame count, size, and last-modified time, with Download and Copy-path buttons. Non-image files (manifests, sidecars) open there too instead of silently downloading. An asset PortOS can't read — a truncated PNG, say — now shows its basic details rather than breaking the whole page.
- **[issue-2928] Show walk previews for every imported run layout, not just one** — the walk viewer found a character's animation runs by looking in a single fixed folder, so an imported character whose source pipeline filed its runs elsewhere showed the same blank "approved ✓" card that #2924 just fixed for redraw directions. Each approved direction is now looked up by where its own approval record says its run lives, so any layout the importer produces animates. Unapproved candidates and in-flight generations still appear as before.
- **[issue-2924] Animate walk directions approved from an imported redraw cycle** — a direction whose approved artwork came from the source pipeline's video-first *redraw* path (rather than a Grok walk clip) showed its "approved ✓" badge with a blank space where the loop preview should be, because there was no walk-run record behind it to render. Those directions now animate from their own packaged cycle: PortOS reads the redraw manifest, prefers its clean transparent strip (falling back to the keyed one), and plays the loop at the cycle's real length and frame rate. The viewer no longer assumes every walk strip is exactly 8 frames, so a 12-frame imported stride plays its full gait instead of being cropped to two thirds; native 8-phase runs are unchanged. A direction whose strip is missing on disk still shows nothing rather than a broken image.
- **[issue-2898] Compile the runtime atlas and publish it to your game (phase 4)** — a finalized walk set can now be compiled into the immutable runtime sprite-sheet (the 10×8 grid of idle, the 8 named gait phases, and a scanner placeholder for all 8 directions, each frame scaled once per direction and pinned to the pivot ground line) and published straight into a managed app's repository. Every compile revalidates the whole evidence chain first — the finalized walk set, every packaged frame, and every locked anchor are checksum-verified — and recompiling the same set is a no-op instead of a new version. Publishing binds a character to a managed app and a repo-relative destination, replaces the atlas atomically, and refuses to clobber a destination that changed outside PortOS since the last publish; an optional code-binding guard verifies (occurrence-counted) that the game source still references the atlas where expected before anything is written. The Sprites tab gains the compile/publish panel with an inline confirmation and a publish history, and sprite walk-run raw frame intermediates are now excluded from backups by default (they regenerate deterministically from the archived source video — the toggle to re-include them is in the Backup tab).
- **[issue-2897] Animate walk cycles from locked anchors (phase 3)** — each locked directional anchor can now be animated: PortOS asks Grok for exactly one walk-in-place video on the character's key-color matte, then packages it locally and deterministically — no further AI involved — into the canonical 8-phase walk strip (frame extraction, chroma un-key and de-spill keyed to the character's selected key color, automatic walk-cycle detection, fixed-scale pivot alignment, and a light/dark/checker contrast review sheet). Review each direction's loop right in the Sprites tab (animated preview), optionally trim frames non-destructively into a versioned loop GIF, and approve per direction; the eighth approval freezes the finalized walk set, which becomes immutable like the reference set. Every run carries a full provenance manifest with per-frame checksums, matching the format the phase-1 importer already understands.
- **[issue-2896] Create characters and build their reference sets with AI (phase 2)** — the Sprites tab can now create a character from a name plus a text description and/or an uploaded design image, generate main-reference candidates through your configured image backend (Codex, Grok, or local), and freeze the approved candidate as the character's immutable identity root. From the frozen main you then derive the 8 directional anchors (front/back/profiles/three-quarters), review each candidate, and lock the set. On lock, PortOS normalizes the art onto a clean key-color square and auto-picks the best chroma key for that character — magenta, green, or blue, whichever sits farthest from the character's own colors (with a manual override limited to the same three keys, and a warning when every key risks clipping). Locked references are server-side immutable: regeneration and replacement are refused, corrections always land as new versions.
- **[issue-2895] Media Gen gains a Sprites section (phase 1: library + import)** — a new Sprites tab lists your game-sprite characters and prop atlas families, browses each one's locked reference set, walk-cycle strips, and published atlas versions (with pixel-perfect previews), and imports the approved production assets from an existing sprite-pipeline checkout in one click. Only final/approved artifacts import — rejected candidates and raw video intermediates stay behind — and every file the source manifests hash-pin is verified on the way in. Reachable from ⌘K and voice ("take me to sprites"). Generation (reference → anchors → walk videos → publish-to-game) lands in the follow-up phases.

## Security

- **The Autofixer UI on `:5560` now honours the PortOS password gate** — when the install is password protected (Settings → Security), the Autofixer sidecar requires the same password before it will list apps, stream PM2 logs, or restart/stop a process. It listens on `0.0.0.0` for tailnet access, so it previously stayed wide open while `:5555` asked for a password. A browser already signed in to the PortOS dashboard on the same host is let straight through (the session cookie is port-agnostic); anything else gets a login prompt, and scripts can use `Authorization: Basic`/`Bearer`. Installs with no password set are unchanged.

- Guard paid-provider API keys against SSRF / key-exfiltration: `streamCompletion` (askService), voice-LLM endpoint resolution, and the local-LLM playground now validate a provider's endpoint through the shared endpoint guard before attaching the `Authorization: Bearer` header, so a mistyped or malicious custom endpoint (including cloud-metadata hosts) never receives the key unless the provider opts in via `allowCustomEndpoint`.
- Bump the `js-yaml` override from 4.2.0 to 4.3.0 (root, server, client) to clear a high-severity advisory where chained YAML merge keys force quadratic CPU consumption.
- Pin server `tar` to 7.5.21 and `engine.io` to 6.6.9 via overrides, clearing a critical node-tar cluster (PAX header file smuggling, parse/decompression DoS, infinite loop on negative entry size) and an Engine.IO polling-transport connection-exhaustion advisory. These live only in `server/package-lock.json`, which is gitignored, so Dependabot never surfaced them.
- **[issue-2848] Clear the remaining `npm audit` HIGH/CRITICAL findings across all three workspaces** — pin `protobufjs` to 7.6.5 and add a `body-parser` 2.3.0 and `sharp` 0.35.3 override in the server, and bring the root and client `brace-expansion` pins up to 5.0.7 / 1.1.16. The `sharp` pin also collapses a stale nested copy that `@huggingface/transformers` carried, so the bundled speech models no longer ship a second, vulnerable libvips build.
