## Brain & CoS API

- **[issue-1835] Brain & CoS list endpoints now support opt-in `limit`/`offset` pagination** — `GET /api/brain/people|projects|ideas|admin|memories` and `GET /api/cos/tasks` previously returned their entire backing store with no bound, while sibling endpoints (`GET /api/brain/links`, `GET /api/memory`) already exposed pagination envelopes. Each of these endpoints now accepts `limit`/`offset` query params: when neither is present they return the exact same shape as before (raw array for the brain collections; the `{ user, cos }` grouped structure for tasks), so every existing client keeps working; when a pagination param is passed they return a bounded `{ <collection>, total, limit, offset }` envelope — and for `/api/cos/tasks` the paginated branch windows each source's `tasks` array, adds a `pagination` block, and drops the full-set derived collections (`grouped`, `autoApproved`, `awaitingApproval`) so the response is genuinely bounded rather than re-leaking the whole queue. `limit` defaults to 50 and is clamped to 500. Shared `isPaginationRequested` / `paginateArray` helpers were added to `server/lib/validation.js` alongside the existing `parsePagination`. (#1835)

## Security

- **`safeUrlFetch` SSRF guard now pins the vetted IP at connect time, closing a DNS-rebinding TOCTOU** — `isPublicHttpUrlSafe` resolved a hostname's DNS to validate it was public, but the subsequent `fetch` re-resolved the same hostname at connect time, so a rebinding host could answer a public address for the check and a loopback/link-local/cloud-metadata address for the actual connection, slipping past the guard. The guard now resolves once and the connection is pinned to that exact validated address via a per-request undici `Agent` whose `connect.lookup` always returns the vetted IP — while Host header and TLS SNI stay the original hostname, so cert validation is unaffected. Every redirect hop is still independently re-validated and re-pinned. (Low real-world exploitability — PortOS is single-user on a private network and URL sources are user-pasted Pinterest boards / stored feeds — but the connect-time re-resolution is now eliminated.) (#1859)
- **Resolved active `npm audit` CVEs in the kokoro-js → protobufjs chain and the client `ws` transitive** — the server's `kokoro-js@1.2.1` pulled `protobufjs <=7.6.0` (GHSA-wcpc-wj8m-hjx6 DoS, GHSA-f38q-mgvj-vph7), and the stale `protobufjs: 7.5.9` override sat *inside* the vulnerable range. Pinned `kokoro-js` to `1.2.0` (the audit-stated fix, still within the code's documented `^1.2.x` compatibility window — `KokoroTTS`/`TextSplitterStream` exports unchanged) and bumped the `protobufjs` override to `7.6.3` in root + `server/package.json`. On the client, `engine.io-client` nested its own `ws@8.20.1` (GHSA-96hv-2xvq-fx4p memory-exhaustion DoS) that the root `ws: 8.21.0` override couldn't reach; added `ws: 8.21.0` to `client/package.json` overrides. Both `npm audit` runs now report 0 vulnerabilities for these chains. (#1823)
- **Backup restore/delete path validation hardened** — `subdirFilter` on `POST /api/backup/restore` was an unconstrained string interpolated into an rsync `--include=${subdirFilter}/***` arg, so `*` could override the filter chain (restoring everything over live data) and `../foo` could traverse out of the snapshot subdir; it now validates against a relative path pattern that rejects wildcard, `..`, and absolute-path segments, with a mirrored defense-in-depth guard in `restoreSnapshot` for non-route callers. `deleteBackup` previously validated `filename.replace(/[.]/g,'')` (so the real dotted filename was never checked, leaving only the `startsWith(backupDir)` guard) — it now validates the raw filename directly and explicitly rejects `..`. (#1822)
- **Settings API no longer leaks third-party API tokens** — `GET` and `PUT /api/settings` redacted only the top-level `secrets` key, so `imageGen.hfToken` (HuggingFace) and `civitai.apiKey` were returned in full on both initial load and after any save. Both responses now run through a single sanitizer that strips those tokens (sibling config preserved), matching how the Settings UI already reads only token *presence* from the dedicated `/image-gen/setup/hf-token-status` and `/loras/auth/civitai` status routes. On-disk values are untouched. (#1821)
- **Vision-test endpoint no longer reads arbitrary files** — `POST /api/providers/:id/test-vision` took `imagePath` straight from the request body, so a `../` traversal or absolute path could have any readable file base64-encoded and forwarded to the configured external vision provider. The image loader now allowlists `imagePath` to a real image basename under `data/screenshots` (via the canonical `resolveScreenshot` path resolver), rejecting traversal and absolute-path escapes before any file is read or forwarded. (#1820)
- **`POST /api/runs` vision screenshots no longer read arbitrary files** — the same unsanitized-`imagePath` gap as #1820 lived in the vendored aiToolkit runner: `screenshots[]` was validated only as `z.array(z.string())` and passed straight to `loadImageAsBase64`, which did `imagePath.startsWith('/') ? imagePath : join(screenshotsDir, imagePath)` — so `{ "screenshots": ["../../../../etc/passwd"] }` (or an absolute path) could have any readable file base64-encoded and forwarded to the configured external vision provider. The untrusted `POST /api/runs` boundary now sanitizes `screenshots[]` via a new `sanitizeScreenshotRefs` helper that `basename`s each entry (collapsing both `../` traversal and absolute-path escapes, including the legitimate absolute paths the Runner UI uploads under `data/screenshots`) and enforces an image-extension allow-list, returning a 400 for any non-image / traversal reference before the run is created. The shared `loadImageAsBase64` loader additionally `basename`s relative refs as defense-in-depth, while still passing trusted in-process absolute paths (e.g. Universe Builder gallery images under `data/images`, already validated by `resolveImageSources`) through unchanged. Kept self-contained — no imports out of the toolkit. (#1870)

## Fixed

- **`useEscapeKey`/`useClickOutside` no longer thrash their global listener on every render** — both hooks kept the caller's `handler`/`onOutside` in the effect dep array, so the `window` keydown/mousedown listener was removed and re-added on every parent render while open. Since most call sites pass a fresh inline arrow (`() => setOpenCommentId(null)`), that meant a full detach+reattach on *every keystroke* in editors like `PipelineManuscriptEditor`/`AnnotatedManuscriptSection` and on every render of `SceneCard`/`WorkEditor`/`UniverseBuilder`. Behavior-preserving; covered by new hook tests. (#1844)
- **Blocking sync I/O removed from request/completion hot paths** — several syscalls ran synchronously on paths that serve live SSE/socket traffic, freezing the event loop for their duration. `checkForTaskCommit` (`agentRunTracking.js`) ran a `spawnSync('git', ['log', …])` on the completion path of every failed CoS run (100ms–2s on a large repo); it now uses the async, hard-timeout `bufferedSpawn`. `buildId.js` `statSync`'d `client/dist/index.html` on every SPA navigation; the freshness check is now throttled to at most once per second (the hot path between checks is a clock comparison, no syscall), so a rebuild re-syncs the served HTML with the on-disk chunks within the throttle window instead of paying a stat per navigation. The avatar (`routes/avatar.js`) and attachment (`routes/attachments.js`) routes swapped `existsSync`/`statSync` for async `fs.access`/`fs.stat` (via a new canonical `pathExists` helper in `fileUtils.js`), and uploads now call the idempotent `ensureDir` directly. (#1828)
- **External API model/metadata fetches now time out instead of stalling callers indefinitely** — `_fetchGeminiModels` (Gemini model refresh in `aiToolkit/providers.js`) made a `fetch` with no `AbortSignal`, unlike every other provider model-fetch in that file, so a hung Google endpoint could block a model-refresh request forever; it now uses the same `AbortSignal.timeout(8000)`. `fetchCivitaiModel` and `searchCivitaiLoras` (`server/lib/civitai.js`) likewise passed no signal, so a Civitai hang stalled LoRA metadata/search callers; both now default to `AbortSignal.timeout(10000)` and accept a caller-provided `signal` override. - **Absent-vs-cleared merge bugs that silently restored values the user just cleared** — three spots conflated "field absent" with "field intentionally cleared to empty".
