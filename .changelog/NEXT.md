# Next Release

## Added

- **FLUX.2-klein image generation on macOS (Apple Silicon).** Three quantized variants ship: 4B SDNQ-4bit (~8 GB @ 512px, recommended), 9B SDNQ-4bit (~12 GB, needs 32+ GB RAM), and 4B Int8 (~16 GB). Runs through PyTorch + diffusers + MPS in a separate `~/.portos/venv-flux2` to avoid colliding with mflux's older torch. `INSTALL_FLUX2=1 bash scripts/setup-image-video.sh` bootstraps the venv (idempotent ‚ÄĒ skips re-install when `Flux2KleinPipeline` already imports). The runner (`scripts/flux2_macos.py`) mirrors `mflux-generate`'s CLI surface so the existing local-image dispatcher just swaps the binary path; live preview frames, metadata sidecar, gallery, and Remix all work unchanged. Stepwise preview decodes latents on a `@torch.no_grad()` callback and writes capped 512-px PNGs so fs.watch + base64 stays cheap. Models are gated on HuggingFace ‚ÄĒ accept the license at https://huggingface.co/black-forest-labs/FLUX.2-klein-4B and export `HF_TOKEN`; the Image Gen page surfaces a yellow banner when the venv is missing or the token isn't set. The legacy `flux2-klein-{4b,9b}` registry entries (which had `broken: 'macos'`) auto-upgrade in place ‚ÄĒ user customizations like a custom name or steps are preserved.
- **Creative Director ‚ÄĒ autonomous long-form video projects** (`/media/creative-director`). Configure a project with a fixed aspect ratio, quality, model, target duration (up to 10 min), style spec, and optional starting image or user-supplied story. A CoS-driven agent writes a scene-by-scene treatment, kicks off 1‚Äď10s segment renders one at a time (continuing from each prior segment's last frame or starting fresh from a new image), evaluates output thumbnails against the style spec via its own multimodal review, and self-perpetuates until the episode is complete. On completion the server auto-creates a video-timeline project and renders the final concat into a single mp4. All segments + the final cut are grouped in an auto-created media collection. New routes (`/api/creative-director/*`), data file (`data/creative-director-projects.json`), preset tables (`server/lib/creativeDirectorPresets.js`), agent prompts (`server/lib/creativeDirectorPrompts.js`), and UI tabs (Overview / Treatment / Segments / Runs).
- **Unified media job queue** (`server/services/mediaJobQueue/`) for image and video gen. Replaces the per-provider 409 BUSY error with a FIFO that accepts every submission immediately and returns `{ jobId, status: 'queued', position }`. A single worker drains serially (image and video share Metal/MLX runtime, so parallelism would OOM anyway). Persists to `data/media-jobs.json`; on boot any job stuck in `running` is reclassified `failed (interrupted by restart)`. SSE on `/api/{video,image}-gen/:jobId/events` covers the full `queued ‚Üí started ‚Üí progress ‚Üí complete/failed` lifecycle so existing UIs keep working without changes. New `/api/media-jobs` lets the UI inspect and cancel queued/running renders.
- **Media model registry at `data/media-models.json`** ‚ÄĒ single source of truth for video and image model definitions, the LTX text encoder, and per-platform defaults. Replaces hardcoded model catalogs in `videoGen/local.js`, `imageGen/local.js`, and `imageVideoModels.js`. Seeded automatically on first run; edit + restart the server to swap models. New `gemma-bf16` (now default, best output quality), `gemma-qat-4bit` (auto-detected if LM Studio installed it), and `gemma-4bit` text-encoder options. Loader resolves `~/...` and prefers a local `localPath` when it exists.
- **`+faststart` mp4 post-processing** ‚ÄĒ every video render is remuxed (no re-encoding, ~hundreds of ms) to move the moov atom to the front of the file. Gallery tiles with `preload="metadata"` now render their poster frame on page load instead of flashing black until clicked.
- **`caffeinate -i -w <pid>`** spawned alongside the LTX Python child on macOS so a 90s+ render isn't aborted by display/system sleep on a laptop. `-w` makes it self-exit when our pid does ‚ÄĒ no manual lifecycle management.
- **Raw stderr/stdout logging from the LTX Python child** with `ūüźć`/`ūüźć-out` prefixes per job. Previously the python child's diagnostic output was filtered for `STATUS:`/`STAGE:` markers and the rest silently dropped, hiding upstream errors like "VAE encoder failed to load" until you ran the CLI manually.
- **`imageStrength` parameter** on the video gen API + service. Plumbed through validation, route, service, and CLI args. mlx_video's actual semantics are inverted from its help text ‚ÄĒ `1.0` preserves the source latent (PortOS default if unspecified), `0.0` fully denoises. Documented inline.
- **1:1 square resolutions** (512√ó512, 768√ó768) added to the Video Gen resolution dropdown.
- **Image upload preview + larger reference-image previews** in Video Gen ‚ÄĒ uploads create object URLs so users see what they uploaded before submit; source / FFLF last-frame / extend-mode previews now render at `max-h-48 object-contain` instead of 48√ó48 thumbnails.

- **Morse code trainer** in POST (`/post/morse`) ‚ÄĒ Koch-method Copy mode (listen ‚Üí type, 90% accuracy unlocks the next letter) and a Send mode that decodes spacebar/touch keying into text. Native Web Audio (no new deps), Farnsworth timing, configurable WPM/tone. Reachable from the POST launcher header, the sidebar, ‚ĆėK, and voice (`ui_navigate "morse"`).
- **Morse trainer side widget** with a binary tree visualization that highlights the live keying path (DAH-left, DIT-right), three reference views (Tree / Length / List), a tap-anywhere practice key, and a real-time decoded log. Spacebar keying is intercepted in capture phase so it doesn't trigger the voice FAB push-to-talk hotkey while on the morse page.
- **Video Gen mode switch + batch queue** (`/media/video`) ‚ÄĒ segmented control selects Text / Image / FFLF (first-frame + last-frame) / Extend (continue from a prior render). Mode-specific input panels (start image, end image, prior-video picker) replace the always-on source image upload. New "Add to queue" button enqueues the current configuration; a batch-queue panel runs jobs serially with per-item status (pending/running/complete/error), inline error messages, and a clear-finished action that drops both successful and errored items. Backend route accepts a `lastImageFile` (gallery-pick) and `mode` field; `--last-image` is plumbed to the LTX Python child (currently advisory until a multi-keyframe wrapper lands).

## Fixed

- **Creative Director stuck projects now self-recover** ‚ÄĒ `completionHook` always re-evaluates project state after any CD task completes (regardless of task kind), so legacy `scene`-kind tasks from before the treatment/evaluate split no longer leave a project frozen. PATCHing a scene to a terminal status (`accepted` or `failed`) also nudges the orchestrator to start the next scene immediately.
- **Creative Director projects auto-resume on server restart.** New `recoverInFlightProjects()` runs at boot (after `initMediaJobQueue` reclassifies running jobs as `failed (interrupted by restart)`): for every project in `planning`/`rendering`/`stitching`, it resets any scenes stuck in `rendering` or `evaluating` (whose in-memory listeners died with the previous process) back to `pending` and fires `advanceAfterSceneSettled` so the orchestrator picks up where it left off. Without this, a pm2 reload mid-render left the project frozen ‚ÄĒ scene status pointed at a dead job, no listener to advance.
- **Creative Director smoke-test fixture** ‚ÄĒ `POST /api/creative-director/smoke-test` (or the "Run smoke test" button on the CD list page) creates a deterministic 3-scene colored-ball project: scene 1 generates a red ball bouncing on white (text-to-video), scene 2 continues from scene 1's last frame with a "blue ball" prompt to verify the i2v honors a prompt-driven color change while preserving motion, scene 3 continues from scene 2 with the same prompt to verify pure i2v continuity. The fixture sets `disableAudio: true` (skips the mlx_video audio pass to roughly halve wall-clock per scene) and `autoAcceptScenes: true` (sceneRunner short-circuits the cognitive evaluator with a synthetic `accepted` evaluation, so a smoke run completes in render time only ‚ÄĒ no Claude in the loop). Adds `disableAudio` and `autoAcceptScenes` as CD project fields and plumbs them through to the queue params.
- **Creative Director scene-retry path unblocked.** `creativeDirectorSceneUpdateSchema` previously stripped `pending` from the agent-callable status enum, but the evaluator-agent prompt template (`creativeDirectorPrompts.js`) and `advanceAfterSceneSettled` (`completionHook.js`) both depend on the agent flipping a scene back to `pending` (with a refined prompt and bumped `retryCount`) to request a re-render. The validation rejected every retry PATCH with `Invalid enum value` and stranded scenes in `evaluating` forever. Schema now accepts the full `SCENE_STATUSES` set and the misleading "agents shouldn't be allowed to flip back to pending" comment is corrected.

- **Image-to-video on macOS now actually preserves the source image as the first frame.** Root cause: PortOS was on `mlx-video-with-audio` 0.1.33, which had a known bug where the VAE encoder failed to load for split-format unified models (LTX-2.3 distilled-Q4) and silently produced gray-fog conditioned frames. Pinned `>=0.1.35` in `scripts/setup-image-video.sh`. The setup script also installed the wrong PyPI package: `mlx_video` (a different unrelated "video loading utilities" package) instead of `mlx-video-with-audio` (the LTX generation backend). Both provide an `import mlx_video` module so the wrong install was silent. Verified end-to-end through the API: pixel-perfect first-frame preservation with coherent motion across all frames.
- **FFLF last-frame conditioning silently produced no output.** PortOS was passing `numFrames - 1` (= 24 for 25 frames) as `--image-frame-idx`, but mlx_video expects a *latent* frame index ‚ÄĒ for LTX that's `floor((videoFrames - 1) / 8)`, max 3 for 25 frames. ffmpeg-conditioning shape check failed and the python child exited 0 with no output mp4. Now computes `lastLatentIdx` correctly.
- **"Continue" / extract-last-frame produced phantom-success logs but no file**, leaving a broken image preview in Video Gen. Two compounded bugs: (1) `-sseof -0.1` was too narrow a seek window ‚ÄĒ for videos with audio (B-frames + AV mux), the last keyframe sits earlier than 100ms before EOF, so ffmpeg returned zero frames while sometimes still exiting 0. Widened to `-sseof -1.0` and added `-update 1`. (2) The success path resolved on exit-code without verifying the output file existed. Now requires `existsSync(framePath)` too.
- **Last-frame extraction now caches** to `data/images/lastframe-<videoId>.png` ‚ÄĒ repeated "Continue" clicks reuse the cached PNG instead of re-spawning ffmpeg. Cuts ~1‚Äď2s per click.
- **Delete operations now log** for both image (`ūüóĎÔłŹ Deleted image: <filename>`) and video (`ūüóĎÔłŹ Deleted video: <filename>`) gallery actions. Previously delete was silent, making it impossible to confirm from the server logs that the click hit the backend.
- **Media model registry tolerates partial user edits.** `loadMediaModels()` now merges parsed `data/media-models.json` over the built-in defaults so a registry missing top-level keys (e.g. `{}` or no `video`) doesn't crash server boot at module import. The loader also catches read errors (permissions, broken symlink) ‚ÄĒ not just JSON parse ‚ÄĒ falling back to defaults so a registry I/O problem can't take down startup. Wrong-type fields are coerced back to defaults too (e.g. `image: {}` or `video.macos: "ltx"` ‚Üí built-in array), so consumers like `getImageModels()` / `buildAppModels()` can't throw at import-time on parseable-but-wrong-shape JSON. `getDefaultVideoModelId()` validates the configured default against the platform's available list ‚ÄĒ typos or `broken`-flagged ids fall back to the first available model with a warning instead of surfacing as "Unknown video model" in the UI. `getTextEncoderRepo()` validates that the resolved entry has a non-empty repo string before passing it to mlx_video. macOS video gen now also validates `model.repo` exists before spawning, surfacing a 500 with a clear error rather than passing `undefined` into spawn args.
- **Last-frame cache validates non-zero size.** A prior ffmpeg crash could leave a 0-byte placeholder at `data/images/lastframe-<videoId>.png`, and the cache hit (`existsSync(framePath)`) would serve it forever as a broken image. `extractLastFrame` now `statSync`s the cached file and falls through to re-extract when it's missing or zero bytes; failed re-extractions also unlink the partial file so the next call retries instead of inheriting the broken cache.
- **Faststart remux works on Windows without risking the rendered video.** `optimizeForStreaming()` did an in-place `fs.rename(tmpPath, videoPath)` which fails on Windows when the destination already exists. The Windows replace strategy now moves the original aside to a `.bak.<uuid>` first, installs the optimized file, and restores the backup on any failure ‚ÄĒ so a transient rename failure (file lock, AV scan, permissions) skips faststart instead of destroying the rendered video. POSIX still uses the single-syscall atomic replace.
- **`caffeinate` sleep-prevention is now strictly macOS-only.** Previous gate (`!IS_WIN`) would also fire on Linux and emit a pointless ENOENT every render. Tightened to `process.platform === 'darwin'`.
- **`scripts/setup-image-video.sh` removes the wrong package before install.** The PyPI package `mlx_video` (unrelated video-loading utility) shadows the LTX backend's `import mlx_video` if previously installed. The script now `pip uninstall --yes mlx_video` before installing `mlx-video-with-audio>=0.1.35`, so upgraders aren't left in an ambiguous state.
- **Form fields stay editable during a video render.** Previously every input was `disabled={generating}`, blocking the user from queueing a different config while a render was in flight. The queue worker captures payloads at enqueue time, so live editing is safe.
- **"Add to collection" popover on media cards no longer gets clipped at the card edge.** `MediaCard`'s outer wrapper had `overflow-hidden` (to clip the image into the card's `rounded-xl`), which was creating a clipping context that swallowed the `AddToCollectionMenu` popover regardless of its `z-index`. Moved the corner-clipping onto the image button itself (`rounded-t-xl overflow-hidden`) so the outer card no longer clips its absolutely-positioned children.
- Video Gen audio uploads (a2v mode) were rejected with "File type not allowed" even for valid MP3/WAV files ‚ÄĒ the multipart parser was missing `fieldname` on the file metadata it passed to the upload filter, so every per-field MIME check silently evaluated to false. Fixed by including `fieldname` in the parser's file metadata, matching multer's API contract.
- `update.sh` no longer hangs after "PortOS restarted" ‚ÄĒ the browser-open script now exits explicitly instead of waiting for Node's undici connection pool to drain.
- `update.sh` now kills the PM2 daemon (not just stops apps) before restarting, fixing crashes where a stale daemon cached a `ProcessContainerFork.js` path from a different project's Yarn PnP zip cache.
- Writers-room storyboard renders queued through Codex now wait their turn instead of erroring with "A Codex generation is already in progress" when you fire a second render before the first finishes. Codex jobs run on their own queue lane, so they don't block (and aren't blocked by) local image or video renders.

## Added

- Video Gen "Chunks" control (1‚Äď8) chains multiple LTX renders into a single longer clip. Each chunk's last frame seeds the next, then ffmpeg stitches them into one output. The individual chunks land in history hidden by default, so the gallery shows just the stitched result.
- Two new LTX-2.3 video models ‚ÄĒ "dgrauet Q4" and "dgrauet Q8" ‚ÄĒ running on a more capable runtime that supports true keyframe interpolation (proper FFLF that respects both start AND end frames), native video extend, and audio-to-video. Install with `INSTALL_LTX2=1 bash scripts/setup-image-video.sh`. The existing notapalindrome models keep working unchanged; pick the new "dgrauet" entries from the model dropdown to use the new pipeline. FFLF on these models has a stage-2 OOM ceiling ‚ÄĒ the server auto-clamps frame count to a memory-safe budget (override via `FFLF_LTX2_PIXEL_BUDGET` env var if you have more RAM than 48 GB).
- "Extend" mode on the new dgrauet runtime now uses a true latent video extension (conditions on the entire source video's motion + visual content) instead of last-frame i2v conditioning. The visible motion stall at scene boundaries is gone ‚ÄĒ extensions continue movement naturally from the source. Legacy notapalindrome models keep using the old chained-i2v workaround.
- New "Audio" mode in Video Gen renders a clip whose motion and on-screen audio sync to an uploaded audio track (WAV / MP3 / M4A). Pick the new mode tile, drop in a 4‚Äď8 second clip, write a directive prompt ("a person dancing to this track in a studio"), and the dgrauet runtime drives the video to the audio. Requires one of the new dgrauet models ‚ÄĒ legacy LTX models don't expose an audio-conditioned pipeline.
