# Unreleased

## Added

- **First-run tutorial for POST cognitive drills.** The timed, stimulus-driven cognitive drills (n-back, digit-span, stroop, schulte-table, mental-rotation, reaction-time) start flashing a stimulus the instant they mount, which is disorienting the first time you meet one. Each drill type now shows a one-time how-to card — a plain-language goal, numbered steps, and the controls — the first time it's encountered, and holds the drill (and its timers) until you tap **Start drill**. The card is config-aware (it names the n-back lag, the digit-span direction, and the reaction-time mode), then never shows again for that type (tracked per type in `localStorage`). Non-timed drills (math/memory, LLM/verbal) are self-explanatory and are not gated.
- **Voice fast-resolution cascade (lower CoS latency, opt-in).** The voice Chief of Staff can now triage each spoken/typed turn through fast client-side tiers before the (slow) server LLM: **tier 1 triggers** resolve navigation commands ("go to tasks", "open the daily log") instantly in the browser with no LLM; **tier 2** runs Chrome's on-device **Gemini Nano** (Prompt API — `window.LanguageModel` / legacy `self.ai.languageModel`) for simple/conversational turns, entirely local and private; **tier 3** is the existing server pipeline (recommend Ollama), which still handles every tool/action turn, personal-data retrieval, dictation, and confirmations. Off by default (existing installs keep all-server behavior); enable under **Settings → Voice → Fast resolution**, which also shows a live on-device-model availability pill with the `chrome://flags` hint. Applies to Web-Speech STT and typed input (whisper/hands-free audio turns stay server-driven); trigger/Nano replies are spoken through the configured TTS and keep barge-in + echo-suppression working. New `llm.fastPath` config (`enabled`/`triggers`/`browserLlm`/`browser.{temperature,topK}`); new client modules `browserLlm.js` (Nano client) and `voiceFastPath.js` (routing decision). Nothing runs a model the user hasn't triggered (no cold-bootstrap).

## Fixed

- **[issue-2540] Stop full-JSONB table loads for pipeline issue id sweeps and bound the multi-account message inbox aggregate.** `listIssueIds` (the conflict-journal orphan-base-hash sweep + tombstone GC) went through `readState()` → `loadAll()`, pulling every issue's full `data` JSONB — up to ~12MB of stage `runHistory` each — into memory only to read the ids; it now sources the store's `SELECT id` projection, with the PG backend filtering tombstones via the mirrored `deleted` column (`listIds({ includeDeleted })`) so the whole-record load is skipped entirely for the default live sweep. Separately, `messageSync.getMessages`'s multi-account inbox spread every message from every account into one array and globally sorted it before slicing a page; it now bounds each account's contribution to the top `offset+limit` by date and merges those heads, so a paged inbox reads `accounts × (offset+limit)` messages into the merge instead of the full cross-account total (the exact `total` count is preserved).
- **[issue-2538] Bound Spotify auth and privacy-broker fetches, and single-flight token refresh.** The Spotify token endpoint (`spotifyAuth.js`) and the BADBOOL / CA-registry broker fetchers (`privacyBrokers.js`) used bare `fetch` with no timeout, so a hung `accounts.spotify.com` could stall an OAuth callback or a history-sync refresh — and a broker source that never responded could hang a user-triggered refresh — indefinitely. All three now go through the shared `fetchWithTimeout` helper with a 15s cap. Separately, concurrent `getAccessToken()` callers that all found the token expired each POSTed their own refresh, and since Spotify rotates the grant the losers persisted a stale token (last-write-wins); refreshes now share one in-flight `refreshPromise` (single-flight), so the grant is rotated exactly once and the guard is released on settle so a failed refresh can retry.
- **[issue-2552] Enlarge above-the-fold mobile touch targets to 44px.** The GoalDetailPanel close button (`p-1` + a 16px icon, ~24px) and the notification bell (40px min) were below the 44×44 touch-target minimum the shell's other chrome controls already meet. The close button now uses `min-h-[44px] min-w-[44px]` centered around its icon (matching the shared `Drawer` control sizing), and the bell's mobile hit area is bumped 40→44px (its `sm:` desktop shrink is unchanged). The notification panel also gains `max-w-[calc(100vw-2rem)]` so its `w-80` width can't overflow a narrow mobile sidebar, and the ⌘K result rows get a `min-h-[44px]` so each is comfortably tappable.
- **[issue-2544] Wiki vault selection now lives in the URL.** The selected Obsidian vault was local `useState`, so a reload, shared link, ⌘K jump, or voice nav always landed you back on the first vault. The vault is now encoded as a `?vault=<id>` search param (the tab route param is preserved when switching tabs), selection is driven from the URL, and a stale/deleted vault id renders a "Vault not found" fallback with a one-click "Show default vault" recovery.
- **POST Digit Span: the form no longer jumps up as digits flash on and off.** During the show phase the visible-digit slot collapsed to zero height in the blank gaps between digits, so the container (padding + `min-h`) oscillated and the recall form/progress bar below shifted up on every flash. The slot now reserves its line box with a non-breaking space in the gaps, keeping the layout steady while digits flash.
- **[issue-2551] Wiki Browse is now a responsive list/detail instead of a fixed two-pane.** The Browse tab always rendered a horizontal two-pane layout with a fixed `w-80` left rail and a magic `calc(100dvh - 220px)` height, so on a phone the folder tree consumed the whole viewport and the note pane was unreachable. It now mirrors the Brain Notes list/detail pattern: a single column on mobile that hides the tree once a note is opened (with a back control to return) and shows both panes from `md+`, laid out on a `grid` that fills its flex parent (`min-h-0`) rather than the fixed viewport-offset height, so a wrapped header no longer clips it.

## Security

- **[issue-2514] Harden AI provider custom endpoints against SSRF / API-key exfiltration.** A provider's `endpoint` is a free-form URL, and model-refresh / API-run paths attached the paid `Authorization: Bearer <apiKey>` and fetched that host with no host checks — a hostile or mistyped endpoint could receive the user's paid LLM key and prompts, or be pointed at a cloud-metadata service (`169.254.169.254`). A new `endpointGuard` now gates every secret-bearing request: loopback/LAN (Tailscale) and known paid-provider hosts are allowed, cloud-metadata hosts are always blocked, and any other host requires the new per-provider **Allow custom endpoint** opt-in. Keyless local-LLM calls (Ollama/LM Studio) are unaffected.
- **[issue-2518] Harden upload path containment and stop leaking absolute FS paths.** The upload/attachment/screenshot routes guarded writes with a bare `resolvedPath.startsWith(DIR)` (no trailing separator, so a sibling like `data/uploads-evil/` string-matched) and the attachment/screenshot responses echoed the absolute on-disk `path` back to the client, leaking the install layout / home-directory username segment. Containment now goes through a shared `isPathInsideDir()` helper (anchored `resolvePath(dir) + sep` prefix, matching `makePathResolver`), responses return only API-relative URLs (`/api/screenshots/<file>`), and the CoS prompt builder resolves those refs back to absolute paths at render time (legacy absolute values pass through unchanged). The `moltbook-claim.mjs` script now logs only the token length + short prefix instead of the full bearer token.
- **[issue-2517] Revalidate the browser navigate URL by resolved IP and by Chrome's actual per-hop connect address.** `POST /api/browser/navigate` only checked the host LITERAL with `isSafeIngestUrl`, so a hostname whose A record rebinds to loopback/link-local/cloud-metadata, or a post-navigate redirect the CDP browser silently follows to a blocked host, slipped past. The route now adds a DNS-resolve pre-check (`assertPublicHttpUrl`) that refuses a hostname resolving to a blocked address before any tab is opened, and drives the navigation through `navigateToUrlPinned` — the same connect-time pin the catalog URL ingest uses — which verifies Chrome's own reported `remoteIPAddress` for the initial nav, every HTTP redirect, and any client-side navigation in the settle window, tearing the tab down and failing closed on a blocked hop. LAN/Tailscale hosts stay intentionally allowed. Residual: a WebSocket to a hostname that DNS-rebinds to a private IP can't be pinned via CDP interception (no `remoteIPAddress` on WS events) — bounded by the single-user trust model and documented in `browserService.ssrfPinRefusalReason`.
- **[issue-2515] Restrict Layered Intelligence custom `cmd` sources to allowlisted binaries (no `shell: true`).** A custom `cmd` source ran its full command string through `shell: true` with only a length/timeout cap, so any config-write path (hand edit, hostile sync payload) gained persistent, unattended RCE on the LI schedule (`; rm -rf ~`, `$(curl … | sh)`, pipes to `sh`). `runShellCommand` now denies the shell by default: the command is parsed and checked against the shared `commandSecurity` binary allowlist (rejecting shell metacharacters and non-allowlisted binaries), then spawned with `shell: false` and parsed args. An operator who genuinely needs a pipeline can set the install-level `settings.layeredIntelligence.trustShellSources` opt-in (off by default) to restore full-shell behavior for that install only.

## Fixed

- **[issue-2537] Serialize message-cache mutators under a per-account write lock.** `refreshMessage` and `updateMessageEvaluations` did their `load → mutate → saveCache` outside the per-account `syncLocks`, so a full-body refresh or an evaluation write racing a concurrent sync (or each other) could clobber the other's update on the shared `data/messages/cache/<accountId>.json` file. All three cache mutators (`syncAccount`, `refreshMessage`, `updateMessageEvaluations`) now route their write region through one per-account tail (`createKeyCachedQueue`), so each awaits the previous and re-reads the freshest persisted state before merging — mirroring `issueWriteTail` in `pipeline/issues.js`. The existing `syncLocks` boolean still rejects a duplicate in-flight sync with 409. Regression tests assert two concurrent mutators both persist.
- **[issue-2543] Cancel stale ⌘K global search responses and clear the Security device-change restart timer.** The command palette's global `search(query)` effect debounced the request but had no in-flight guard, so a slow response for an older query could resolve after the user typed on and overwrite the newer query's results (or flip the loading spinner off prematurely) — the sibling catalog-search effect already used a `cancelled` flag. The global-search effect now mirrors that `cancelled` guard so only the latest query's response applies. Separately, `Security.jsx`'s device-change handler scheduled `setTimeout(startMedia, 100)` with no clear on unmount or before rescheduling, so rapid device switches stacked multiple `startMedia()` calls and a navigation-away could fire the restart into an unmounted component; the timer id is now held in a ref, cleared before each reschedule, and cleared in an unmount cleanup effect. A regression test asserts a stale search response can't clobber the current query's results.
- **POST session run page crashed (React error #130) when transitioning into a cognitive drill.** `DrillTransition.jsx`'s `DOMAIN_ICONS` map was missing a `cognitive` entry, yet `DRILL_TO_DOMAIN` maps every cognitive drill type (n-back, digit-span, stroop, schulte-table, mental-rotation, reaction-time) to the `cognitive` domain — so finishing a drill whose successor was a cognitive drill (e.g. a Doubling Chain followed by a cognitive drill) resolved the transition-card icon to `undefined` and blanked the run page with a minified React error. Added the `cognitive: Brain` icon and hardened the lookup to fall back to `ChevronRight` for any unmapped domain, so a future domain gap degrades gracefully instead of crashing. A regression test renders the transition for every `DRILL_TO_DOMAIN` type.
- **[issue-2549] Enforce the Windows `.exe` basename contract in the CoS Runner allowlist test.** The `allowedCommands.test.js` case for a Windows absolute path (`C:\\...\\claude.exe`) ended on `expect(typeof result).toBe('boolean')`, which passes for any boolean — so on POSIX CI the Windows path handling was never actually asserted and a regression would go unnoticed. It's replaced with concrete assertions: a POSIX-style `.exe` path (`/opt/tools/claude.exe`) that exercises the basename-then-strip logic deterministically on every OS (allowed), a matching rejection for a non-allowlisted `.exe` (`/opt/tools/bash.exe`), and a per-platform assertion for the real backslash form that expects `true` on `win32` and `false` on POSIX (where `path.basename` doesn't split backslashes). Both new assertions were verified to fail against a broken implementation (dropping the `.exe` strip; forcing `win32` basename on POSIX). Test-only change — no runtime behavior change.
- **[issue-2536] Catch unhandled rejections from Telegram handlers and health/probe timers.** Several async boundaries were fire-and-forget: the Telegram client dispatched `onText`/`emitter.on` handlers without awaiting or catching, `telegram.js` ran `setInterval(healthCheck)` and a couple of sync-context `bot.sendMessage(...)` calls without a `.catch`, and `instances.js` scheduled `probeAllPeers()` from `setTimeout`/`setInterval` without one — so a rejected handler or a failed fs/network read leaked an unhandled promise rejection (process-killing on Node ≥15). The client now routes every handler invocation through a `runHandler` wrapper that logs a single emoji-prefixed line on rejection; the health-check interval and both probe timers wrap their async callback in `.catch`; and the `/start`, `/help`, and unauthorized-notice sends now await or `.catch` their promise. A vitest suite drives the client's `onText` and `message` paths with a throwing handler and asserts no `unhandledRejection` escapes.
- **[issue-2519] CoS handlers no longer flash a false-success toast after a failed API call.** Several Chief of Staff actions (`handleForceEvaluate` on the CoS page, the Tasks tab "Run Now" button, and the Config tab's Save + autonomy-level change) `.catch`-swallowed the request rejection and then unconditionally toasted success and mutated optimistic UI — so a network/server error still read as "done." Each now awaits inside try/catch and only toasts success / advances state after the request resolves: a failed Save keeps the editor open, and a failed autonomy-level change reverts the optimistic form params. Errors surface via a single `toast.error` (the calls already pass `{ silent: true }`).
- **[issue-2520] Pass `{ silent: true }` on client API calls that own a custom error toast.** Several call sites routed a request through the auto-toasting `request()` helper AND showed their own `toast.error` in `.catch`, so a failure surfaced two stacked toasts. The music managers (Artists/Albums/Tracks load + create/update/delete), MusicVideo (list/create/delete/analyze/plan/add-scene/delete-scene/reorder), and DataDog (load/save/delete/test) now pass `{ silent: true }` on those calls; CharacterSheet's local `charGet`/`charPost`/`charPut` helpers are silent-by-default (every caller owns its own catch toast or the `loadError` banner) with an opt-out. Each failing action now shows a single toast.
- **[issue-2510] MidiPianoRoll: clear the stale hover overlay when the transcription data changes.** The piano-roll kept its hovered note/chord identity (objects from the previous parse) when the `data`/`chords` props swapped on a mounted instance, so a re-transcription or Retry while a hover was active and the pointer sat still would keep stroking the old note at coordinates mapped into the new file until the next pointer move or Escape. A `useEffect` now drops the hover identity whenever `data`/`chords` change.
- **[issue-2513] Block loopback/metadata hosts in Layered Intelligence custom `http` sources.** `fetchHttpSource` used a bare `fetch` with only an `http(s)` scheme check, so a hand-edited config could point a custom source at `127.0.0.1`, `169.254.169.254`, or `metadata.google.internal` and exfiltrate up to 8KB into the reasoner prompt. It now routes through the shared SSRF-guarded `fetchPublicText` (default posture): loopback, link-local, and cloud-metadata hosts are blocked and redirects are revalidated, while LAN/Tailscale hosts stay intentionally allowed for this single-user tool.
- **[issue-2516] Database sync: remove shell interpolation from the dump import.** The active→inactive backend sync built a `bash -c "sed … \"${dumpFile}\" | psql … -U ${pgUser} -d ${pgDb}"` string that interpolated the dump path, port, user, and database name into a shell command — any shell metacharacter in those values (env-derived or surfacing from the export pipeline) was injectable into `psql`. The import now streams the dump through `spawn('psql', argv)` with no shell: the dump is read and piped to psql's stdin, and the pg17-only directives (`\restrict`/`\unrestrict`/`SET transaction_timeout`) the old `sed` stage stripped are filtered in-process. No user- or env-derived value ever reaches a shell string.
