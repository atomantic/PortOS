## Added

- **Kokoro voice preview button in Settings → Voice** — new ▶ button next to the voice dropdown lets you audition any Kokoro (or Piper) voice without saving. Hits `POST /api/voice/test` with an optional `voice` override that temporarily swaps the voice for the test synth call — the saved config is untouched. Per-button spinner while rendering; button self-disables when another preview is in flight.
- **Human-readable Kokoro voice labels in Settings → Voice** — voice picker options now render as "American Female — Heart (A)" / "British Male — Fable (C)" instead of the raw `af_heart (Female) — A` form. Accent is derived from the voice's `language` field (`en-US` → American, `en-GB` → British) and the name after the underscore is title-cased. Trait icons (❤️, 🔥, 🎧) still prefix the label.
- **STT engine picker in Settings → Voice** — surfaces the existing `stt.engine` config option in the UI. Auto-disables the Web Speech API choice on browsers that don't expose `SpeechRecognition` (Firefox, older Safari) with a "not supported here" marker. Whisper-specific fields (model, endpoint, CoreML toggle) now hide when the engine is Web Speech, keeping the form focused.

## Changed

- **Default STT engine flipped from Whisper to Web Speech API** — fresh installs and the sample `data.sample/settings.json` now default to `stt.engine: 'web-speech'` (browser-native, zero-latency, no server process). `reconcile()` in `server/services/voice/bootstrap.js` now short-circuits when STT is Web Speech: stops any leftover `portos-whisper` PM2 app and skips binary/model verification and download so users who stick with the default don't pay for the whisper.cpp model or process. Health probes skip the whisper HTTP check in this mode and report a synthetic `web-speech: { ok: true, state: 'browser-native' }` status so the Settings page no longer shows whisper as "down" when it's intentionally off. Existing installs keep their saved `stt.engine` value — flip it via Settings → Voice → STT Engine.

- **Web Speech API STT mode for voice assistant** — new `stt.engine: 'web-speech'` config option uses the browser's native SpeechRecognition API for speech-to-text instead of whisper.cpp. Transcription runs entirely in-browser with zero network latency — final transcripts are sent via the existing `voice:text` path so no server changes needed. Hands-free toggle and VAD meter hidden in this mode (browser handles VAD natively). Interim transcripts appear live in the conversation panel. Enable via Settings → Voice → STT Engine.
- **Local voice Chief-of-Staff with hands-free VAD and opt-in tool calling** — new voice stack under `server/services/voice/` and `client/src/{services,components}/voice/` runs entirely on-device: Whisper (STT) + Kokoro or Piper (TTS) + LM Studio (LLM). Two capture modes: push-to-talk (click once to listen, click again to send) and **hands-free continuous mode** (default) that auto-submits on a ~600ms pause and lets you barge in over the bot's own voice. Hands-free is built around an AudioWorklet streaming raw PCM to a thin energy-based VAD — thresholds auto-calibrate to the user's ambient noise floor on startup, a pre-roll ring buffer avoids clipping the first syllable, a TTS echo-tail cooldown filters speaker→mic reverb, and a runaway-speech watchdog unjams VAD if silence never registers. Whisper requests now bias the decoder with a PortOS vocabulary prompt so proper nouns ("brain inbox", "Chief of Staff", "Tailscale") don't drift to homophones.
- **Configurable CoS personality for the voice assistant** — Settings → Voice exposes a personality editor (name, role, traits, speech style, custom prompt). The system prompt is composed live per turn from those fields plus the list of available tools. Toggle **Enable tools** (off by default; needs a tool-use-trained model like Qwen2.5-7B or Hermes-3) to let the CoS actually do things. First tool shipped: `brain_capture` — routes any "remember…", "add…", "note…" style request into the existing brain inbox via `captureThought()`. The pipeline runs up to 3 tool-call rounds per turn and streams text deltas to TTS between iterations so you hear the CoS narrate what it's doing.
- **Voice widget on every page** — floating conversation card (bottom-right) with live mic-level meter, stage indicator (`handsfree` / `capturing` / `thinking` / `speaking`), mode toggle, interrupt button, text-entry fallback, and scrolling transcript. Hotkey (default Space) toggles listening. Settings → Voice also gets STT/TTS engine pickers, Whisper model picker, Kokoro voice picker (dozens of voices: `af_*` female, `am_*` male, `bf_*`/`bm_*` British), CoreML encoder toggle, and one-click "Test voice".
- **MortalLoom iCloud sync** — new Settings tab (Settings → MortalLoom) lets PortOS share Goals and Meatspace data with the MortalLoom iOS/macOS app by reading from and writing to the app's iCloud ubiquity container (`~/Library/Mobile Documents/iCloud~net~shadowpuppet~MeatSpaceTracker/Documents/MortalLoom.json`). When enabled, PortOS treats the iCloud file as the source of truth for alcohol drinks, nicotine entries, blood tests, body composition, epigenetic tests, eye exams, and goals — and every add/update/remove flow mirrors back to it so edits on either side stay in sync. Non-destructive one-shot "Import from MortalLoom" button for adopting the iCloud data into local PortOS stores by id-dedup. New `server/services/mortalLoomStore.js` exposes tight primitives (`readStore`, `updateStore`, `mlPush/mlPatchById/mlRemoveById/mlReplace`, `mlArrayIfEnabled`, `mlIdAtDateIndex`) used by `meatspaceHealth`, `meatspaceAlcohol`, `meatspaceNicotine`, and `identity`.
- **MortalLoom promo banner on Meatspace + Goals pages** — dismissible banner (stored in `localStorage` as `mortalloom-banner-dismissed-v1`) linking to the App Store (`id6760883701`) and to the new Settings tab for iCloud sync configuration.

## Fixed

- **Digital Twin build errors — missing Soul-prefixed API exports** — components used `api.createSoulDocument`, `api.runSoulTests`, etc. but `apiDigitalTwin.js` only exported `DigitalTwin`-prefixed names. Added 10 Soul aliases so the build resolves all imports.

- **POST drill scoring paired every answer to the first prompt** — `llmResponseSchema` in `postValidation.js` didn't list `questionIndex`, so Zod's default strip mode dropped the field on its way to the scorer. The scorer's `r.questionIndex ?? i` fallback then resolved to the response-array index — always 0 for the single-response submits the trainer makes — and every answer was evaluated against the FIRST prompt of the drill. Affected bridge-word, idiom-twist, double-meaning, and every other LLM-scored POST drill (the v1.43.0 client-side fix wired up `questionIndex` on the request but it never reached the scorer). Schema now declares `questionIndex` and a regression test in `postValidation.test.js` covers the round-trip.
- **Compound chain rejected the natural shorthand** — typing just the *other half* of a compound (e.g., `hose`, `pit`, `work` for the root `fire`) was scored as invalid because the local scorer required the user's input to literally include the root word. Players had to type `firehose` / `firepit` / `firework` in full, which slowed them down and produced misleading "0 valid compounds" scores. Scorer now accepts both the full compound and the half — bare root word still rejected, and the "you could also have said" suggestions correctly suppress examples the user already covered via either form. UI placeholder updated to "Type the full compound or just the other half...".
