## Added

- **Scheduled tasks and the new-task form can pin a thinking-effort level.** Alongside the existing provider/model overrides, a "Thinking Effort" select appears when the pinned provider is Claude Code or Codex (Claude: lowâ†’max; Codex: minimalâ†’ultra). ## Fixed

- **[issue-2507] CoS app-improve / self-improvement tasks no longer stall waiting on a Copilot review that never arrives.** These scheduled tasks pin no `reviewers`, so the agent-prompt builder resolved the review-loop reviewer straight to the hardcoded `copilot` default â€” and on installs without GitHub Copilot code review enabled the agent requested a review that never came and idled out in the merge/review-loop phase. The two reviewer-resolution sites in `agentPromptBuilder.js` now thread the install's configured **Code Review Defaults** (`settings.codeReview.reviewers`) as the fallback â€” the same source the scheduler and review-loop follow-up already honor â€” so a user who sets `claude,codex,ollama` there has their app-improve tasks use it. With nothing configured, behavior is unchanged (copilot); a settings read error degrades to the hardcoded default.
- **Codex agents no longer idle out when a new codex release is available.** On every interactive startup codex checks for a newer release and, if one exists, renders a blocking "âś¨ Update available!" modal whose highlighted default is "1. Update now (runs `brew upgrade --cask codex`)". A headless CoS/TUI agent has no human to steer it: the paste handshake's submit-Enter landed on "Update now", codex shelled out to a ~100 MB brew upgrade, and the session ended at "đźŽ‰ Update ran successfully! Please restart Codex." â€” a dead box that never processed the prompt, so the agent idled out reporting "prompt likely never submitted" (observed repeatedly on gpt-5.6-sol codex-tui runs). PortOS now spawns codex with `-c check_for_update_on_startup=false` across every codex spawn path â€” interactive TUI, the two `codex exec` CLI builders, and the vision-caption path â€” via a shared `buildCodexStartupArgs` helper (alongside the existing `buildEffortArgs`), so the input box comes up live and no run wastes startup time on an unattended upgrade it can't complete. The override is skipped when a provider already pins the key.
- **Local image gen: fixed a black band at the bottom of Z-Image-Turbo (and other diffusers-runner) outputs.** The shared runner helper unconditionally enabled VAE *tiling*, whose overlap-blend leaves a seam / dark strip at a content-dependent point near the bottom of the image â€” most visible at tall aspect ratios like 576Ă—1024 (9:16). Tiling only saves memory on very large decodes and the small-to-mid resolutions PortOS renders never need it (the same reason both runners already disable tiling for image-to-image). VAE tiling is now applied only above a ~6.5 MP area floor that no standard or experimental preset reaches, so ordinary renders decode in one clean pass; pathologically large custom dimensions still get tiling as an out-of-memory safety valve. VAE slicing and attention slicing (which never affect spatial layout) stay on for every render.
- **AI Providers warns before enabling the Grok Build CLI/TUI.** The Grok harness uploads your entire working repo to xAI (GCP) as it works. - **Layered Intelligence no longer wedges on a stale API-only provider.** Any install that ran the Layered Intelligence loop on an api provider (ollama/lmstudio/kimi) before it became agent-backed (#2322) had that provider carried into the per-app override by migration 184 â€” where it now outranks the global Schedule pin at spawn. The loop now self-heals: an api-only per-app provider falls back to a CLI/TUI schedule pin (adopting its matched model), so the Schedule-page selection finally takes effect. - **Layered Intelligence no longer silently drops a proposal when a local model's JSON is slightly malformed.** A reasoning run pinned to a smaller/local model (e.g. Ollama qwen) would find a real gap, then write an almost-valid `.agent-done` envelope â€” wrapped in ```json fences, trailed by prose, or with literal newlines/tabs pasted into the long markdown body â€” that the strict sentinel parser rejected, recording "unparseable-response" and filing nothing.
